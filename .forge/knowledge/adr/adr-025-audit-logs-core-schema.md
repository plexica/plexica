# ADR-025: Audit Logs Placement in Core Schema vs Tenant Schema

> Architectural Decision Record documenting a deliberate, justified exception
> to the schema-per-tenant pattern (ADR-002) for the `audit_logs` table.
> Audit logs are placed in the core (shared) schema with mandatory safeguards
> to preserve tenant isolation while enabling cross-tenant compliance reporting.

## Context

ADR-002 established **schema-per-tenant** as Plexica's database multi-tenancy
strategy. Under this pattern every tenant's data lives in an isolated
PostgreSQL schema (`tenant_{slug}`), so a missing `WHERE tenant_id = ...`
clause in a query is _structurally impossible_ — the search path simply does
not include other tenants' schemas. This is a core security guarantee
(Constitution Article 1.2 §2, Article 5.2 §4).

Spec 008 (Admin Interfaces) introduces an **Audit Log** system:

- **FR-007 (Tenant Admin Audit Log)**: Tenant admins view audit logs scoped
  to their own tenant — user actions, configuration changes, plugin events.
- **FR-006 (Super Admin Portal)**: Super Admins view audit logs across _all_
  tenants in a single query for compliance investigation, security incident
  response, and cross-tenant usage analytics.

These two requirements are in direct tension:

1. **Schema-per-tenant placement** satisfies FR-007 (each tenant sees only its
   own logs, enforced by the schema boundary) but makes FR-006 extremely
   difficult — Super Admin cross-tenant queries would require `UNION ALL`
   across potentially hundreds of `tenant_{slug}.audit_logs` tables. This is
   prohibitively slow, operationally complex (dynamic SQL generation for an
   unknown number of schemas), and breaks Prisma's query model (Art. 3.3).

2. **Core schema placement** satisfies FR-006 trivially (single-table query
   with optional `tenant_id` filter) but weakens the schema-level isolation
   guarantee — if any query path omits the `tenant_id` predicate, audit data
   from one tenant could leak to another tenant's admin.

This tension was flagged as **W-302** during `/forge-analyze` of Spec 008
plan.md §4.1.

### Why Audit Logs Are Different

Audit logs are **compliance infrastructure**, not tenant business data:

- They record _platform events_ (who did what, when, on which tenant) for
  regulatory, security, and operational purposes.
- Their primary consumers are the Super Admin (cross-tenant) and compliance
  systems, not the tenant's own application logic.
- Tenant admins read them for accountability, but never write to them
  directly — logs are append-only, generated by platform services.
- In compliance frameworks (SOC 2, ISO 27001), audit logs are typically
  centralized, not partitioned by tenant.

## Options Considered

### Option A: Core Schema with Mandatory Safeguards (Chosen)

Place `audit_logs` in the `core` schema as a **documented exception** to
ADR-002, with explicit safeguards to enforce tenant isolation at the
application and database level.

- **Pros**:
  - Cross-tenant queries (FR-006) are trivial — single-table, single-index scan
  - Compliance reporting and security incident investigation work naturally
  - Single migration path; no schema-propagation overhead
  - PostgreSQL RLS provides defense-in-depth isolation at the database level
  - Consistent with how audit/compliance infrastructure is modeled in SOC 2
    frameworks

- **Cons**:
  - Breaks the schema-level isolation guarantee from ADR-002 for this table
  - Tenant isolation relies on application logic + RLS rather than schema boundary
  - Requires discipline: every new query path must include `tenant_id`
  - Any bypass of `AuditLogRepository` is a potential data leak

- **Effort**: Low (standard table in core schema)
- **Risk**: Medium — mitigated by the 5 mandatory safeguards below

---

### Option B: Tenant Schema Placement (Rejected)

Place `audit_logs` in each `tenant_{slug}` schema, matching ADR-002.

- **Pros**:
  - Full schema-level isolation — impossible to leak across tenants
  - Consistent with every other tenant data table

- **Cons**:
  - **FR-006 is effectively impossible**: Super Admin cross-tenant queries
    require dynamically-generated `UNION ALL` across all tenant schemas.
    At 500 tenants, this means 500-way UNION — Prisma cannot express this,
    raw SQL is fragile and slow, and PostgreSQL's query planner degrades
    severely with large UNIONs.
  - Compliance incident response becomes O(n) in tenant count instead of O(1)
  - No single index for cross-tenant searches (e.g., "find all actions by
    user X across all tenants they have access to")

- **Effort**: Medium (standard migration, but Super Admin queries are very hard)
- **Risk**: High — FR-006 cannot be implemented performantly

**Rejected** because the cross-tenant query impossibility makes the Super Admin
compliance portal unimplementable within acceptable performance bounds.

---

### Option C: Separate Audit Database (Deferred)

Create a dedicated PostgreSQL database (or time-series database like
TimescaleDB) for all audit logs.

- **Pros**:
  - Complete separation of audit infrastructure from application data
  - Could use purpose-built time-series storage (compression, retention policies)
  - No risk of audit queries impacting application database performance
  - Natural fit for compliance archival requirements

- **Cons**:
  - Significant infrastructure cost (new database to provision, monitor, back up)
  - Cross-database transactions not possible in PostgreSQL — audit log writes
    would be eventually-consistent with the actions they record
  - New dependency (TimescaleDB or separate PG instance) requires ADR + constitutional amendment
  - Overkill for current scale (<1000 tenants, <100 concurrent admins)

- **Effort**: High (new infra, connection management, eventual consistency handling)
- **Risk**: Low isolation risk, but high operational complexity

**Deferred** to Spec 008 Phase 2 or a dedicated audit infrastructure spec.
If audit volume exceeds 10M rows/month or compliance requirements mandate
separate storage, this option should be revisited.

## Decision

**Place the `audit_logs` table in the core (shared) schema as a deliberate,
documented exception to ADR-002 (schema-per-tenant).** This is the **only**
table with this exception.

This decision is contingent on full implementation of the 5 mandatory
safeguards below. If any safeguard cannot be implemented, this decision must
be revisited.

### Mandatory Safeguards

These safeguards are **non-negotiable** — they are conditions of this ADR,
not suggestions:

#### 1. Single Access Path: `AuditLogRepository`

All access to the `audit_logs` table MUST go through the `AuditLogRepository`
class. No raw queries, no direct Prisma `audit_logs` model access outside
this class.

```typescript
// File: apps/core-api/src/modules/audit/audit-log.repository.ts
export class AuditLogRepository {
  // ALL audit_logs queries are here — no exceptions
}
```

Any module that needs audit data MUST depend on `AuditLogRepository` or the
`AuditLogService` that wraps it. Direct `prisma.auditLogs.findMany()` calls
from outside this module are a security violation.

#### 2. Required `tenantId` Parameter for Tenant-Scoped Queries

Every `AuditLogRepository` method that serves tenant admin requests MUST
accept `tenantId` as a **required** parameter (not optional, not nullable).
The `WHERE tenant_id = :tenantId` clause is enforced by the method signature.

```typescript
// ✅ CORRECT: tenantId is required — cannot be forgotten
async findByTenant(tenantId: string, filters: AuditLogFilters): Promise<AuditLog[]> {
  return this.prisma.auditLogs.findMany({
    where: { tenantId, ...this.buildFilters(filters) },
    // ...
  });
}

// ❌ FORBIDDEN: optional tenantId — allows accidental cross-tenant access
async find(filters: { tenantId?: string; ... }): Promise<AuditLog[]> { ... }
```

#### 3. Explicit Super Admin Cross-Tenant Methods

Methods that query across tenants MUST be:

- Named explicitly to indicate cross-tenant scope (e.g., `findAllAcrossTenants`,
  `searchCrossTenant`)
- Gated by a Super Admin role check at the service layer
- Never callable from tenant-scoped controller routes

```typescript
// Only called by SuperAdminAuditController, which validates SUPER_ADMIN role
async findAllAcrossTenants(filters: CrossTenantAuditFilters): Promise<AuditLog[]> {
  // No tenantId filter — intentional, for Super Admin only
  return this.prisma.auditLogs.findMany({
    where: this.buildCrossTenantFilters(filters),
    // ...
  });
}
```

#### 4. PostgreSQL Row-Level Security (Defense-in-Depth)

Add an RLS policy on `audit_logs` as a database-level backstop:

```sql
-- Enable RLS on audit_logs
ALTER TABLE core.audit_logs ENABLE ROW LEVEL SECURITY;

-- Tenant-scoped access: filter by current tenant context
CREATE POLICY tenant_isolation ON core.audit_logs
  USING (tenant_id = current_setting('app.current_tenant_id')::uuid);

-- Super Admin bypass: separate policy for Super Admin sessions
CREATE POLICY super_admin_access ON core.audit_logs
  USING (current_setting('app.is_super_admin', true)::boolean = true);
```

The tenant context middleware already sets `app.current_tenant_id` via
`SET LOCAL` on each request. For Super Admin routes, additionally set
`app.is_super_admin = true`.

RLS is defense-in-depth — even if a bug bypasses the `AuditLogRepository`
abstraction, PostgreSQL will enforce tenant filtering at the database level.

#### 5. Code Review Gate

Any new method added to `AuditLogRepository` MUST be reviewed for tenant
isolation compliance. The review checklist:

- [ ] Does the method include `WHERE tenant_id = :tenantId` if it serves
      tenant admin requests?
- [ ] If it's a cross-tenant method, is it explicitly named and gated by
      Super Admin role check?
- [ ] Does the method avoid exposing raw query builders or Prisma model
      access to callers?

This requirement MUST be documented in the `AuditLogRepository` class header
comment and enforced during code review (Constitution Article 4.2).

## Consequences

### Positive

- **Cross-tenant audit queries are simple**: Super Admin compliance portal
  queries a single table with standard filters — no UNION ALL, no dynamic SQL,
  no O(n) schema enumeration.
- **Compliance-ready**: Centralized audit log matches SOC 2 / ISO 27001
  expectations for audit infrastructure.
- **Single index path**: Cross-tenant searches (e.g., "all actions by user X")
  use one composite index instead of N separate indexes across N schemas.
- **Migration simplicity**: One migration, one table, one schema — no
  propagation to all tenant schemas required.

### Negative

- **Weakened isolation for this table**: Tenant isolation for `audit_logs`
  depends on application logic + RLS rather than the schema boundary. This is
  a strictly weaker guarantee than schema-per-tenant.
- **Requires ongoing discipline**: Every new query path must be reviewed for
  tenant isolation. A single missed `tenantId` filter is a data leak.
- **Sets a precedent**: Future developers may cite this exception to justify
  placing other tables in core schema. This ADR explicitly states: **this is
  the only table with this exception.** Any additional exceptions require a
  new ADR with equivalent safeguards.
- **RLS complexity**: PostgreSQL RLS adds a layer of configuration that must
  be maintained and tested. `SET LOCAL` for tenant context must be verified
  in integration tests.

### Neutral

- If audit volume grows beyond the capacity of a single PostgreSQL table
  (>100M rows), table partitioning by `created_at` (monthly partitions) can
  be added without changing the application layer. This is a standard
  PostgreSQL operational concern, not an architectural change.
- The `AuditLogRepository` abstraction makes future migration to a separate
  audit database (Option C) straightforward — only the repository internals
  change; all callers are unaffected.

## Implementation Notes

### Database Migration

```sql
-- Migration: YYYYMMDDHHMMSS_create_audit_logs.sql
-- Schema: core (shared)

CREATE TABLE core.audit_logs (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id     UUID NOT NULL REFERENCES core.tenants(id),
  actor_id      UUID NOT NULL,
  actor_type    VARCHAR(20) NOT NULL,  -- 'user', 'system', 'plugin'
  action        VARCHAR(100) NOT NULL, -- 'tenant.settings.updated', 'user.created', etc.
  resource_type VARCHAR(50) NOT NULL,  -- 'tenant', 'workspace', 'plugin', 'user'
  resource_id   UUID,
  metadata      JSONB DEFAULT '{}',
  ip_address    INET,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Indexes for tenant-scoped queries (FR-007)
CREATE INDEX idx_audit_logs_tenant_created
  ON core.audit_logs (tenant_id, created_at DESC);

CREATE INDEX idx_audit_logs_tenant_action
  ON core.audit_logs (tenant_id, action, created_at DESC);

-- Index for cross-tenant queries (FR-006)
CREATE INDEX idx_audit_logs_created
  ON core.audit_logs (created_at DESC);

CREATE INDEX idx_audit_logs_actor
  ON core.audit_logs (actor_id, created_at DESC);

-- Row-Level Security
ALTER TABLE core.audit_logs ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON core.audit_logs
  USING (tenant_id = current_setting('app.current_tenant_id')::uuid);

CREATE POLICY super_admin_access ON core.audit_logs
  USING (current_setting('app.is_super_admin', true)::boolean = true);
```

### Test Requirements

The following integration tests are **mandatory** before this table ships:

1. **Tenant isolation test**: Insert audit logs for tenant A and tenant B.
   Query via `AuditLogRepository.findByTenant(tenantA)` — assert tenant B
   logs are NOT returned.
2. **RLS enforcement test**: Set `app.current_tenant_id` to tenant A, then
   run a raw `SELECT * FROM core.audit_logs` — assert only tenant A rows
   are returned.
3. **Super Admin cross-tenant test**: Set `app.is_super_admin = true`, query
   via `findAllAcrossTenants()` — assert logs from all tenants are returned.
4. **Missing tenant context test**: Attempt a query without setting
   `app.current_tenant_id` — assert it returns zero rows (RLS blocks access).
5. **Role gate test**: Call `findAllAcrossTenants()` from a non-Super-Admin
   context — assert it throws `ForbiddenException`.

### Design Decisions Summary

1. **Two distinct shells**: Super Admin (`/super-admin`) and Tenant Admin (`/admin`) are separate React apps (or route trees) with independent sidebar navigation. Super Admin has no tenant context; Tenant Admin inherits the active tenant's theme. Per NFR-003/NFR-004.
2. **Information-dense layout**: Admin interfaces use compact spacing (`--space-3` card padding, `--space-2` table cell padding) per design system principle §1.3. Tables are the primary data pattern; cards reserved for dashboards and summaries.
3. **Wizard for provisioning**: Tenant creation uses a 3-step wizard with real-time provisioning progress (NFR-008). All other CRUD operations use standard forms within modals or inline panels.
4. **Destructive actions require confirmation**: All suspend, delete, and deactivate actions use a `DestructiveConfirmModal` with typed confirmation (type the tenant name to confirm deletion). Per design system anti-pattern §11.
5. **Client-side validation on blur**: Per NFR-005 and design system convention (Web SPA rule), all forms validate on field blur. Error messages appear inline below the field. Server-side errors mapped to fields per Art. 6.2 format.
6. **Responsive strategy**: At 768px, sidebar collapses to hamburger overlay. Data tables gain horizontal scroll. Action buttons collapse into a `MoreVertical` menu. Dashboard metric cards stack vertically.

- Form validation: `aria-invalid="true"` + `aria-describedby="slug-error"` on invalid fields
- Cancel button: no confirmation needed (no data saved yet)
- Focus management: focus moves to first field of each step on navigation

## 6. Accessibility Checklist (WCAG 2.1 AA)

Applied to ALL 19 admin screens:

### Perceivable

- [x] All images have `alt` text (tenant logos: `alt="Acme Corp logo"`; decorative icons: `alt=""`)
- [x] Color is not the only way to convey information (status badges always include text label + color; role editor uses text labels alongside checkbox state)
- [x] Text contrast ≥ 4.5:1 for body text, ≥ 3:1 for large text (all token pairs verified — see §5 contrast table + design-system.md §Contrast Verification)
- [x] Interactive element contrast ≥ 3:1 (buttons, links, badges all use verified token pairs)
- [x] Text resizable to 200% without horizontal scrolling (admin layout uses `rem` units; tables gain horizontal scroll at 768px — this is acceptable per WCAG for data tables)
- [x] No content flashes more than 3 times per second (no animations exceed this)

### Operable

- [x] All functionality accessible by keyboard alone (all buttons, links, modals, dropdowns, table sorting, pagination keyboard-navigable)
- [x] No keyboard traps (modals have Escape; overlay sidebar has Escape; tab order is linear)
- [x] Skip navigation link provided ("Skip to main content" on every admin page)
- [x] Tab order is logical and matches visual order (left-to-right, top-to-bottom within each panel)
- [x] Focus indicator visible on all interactive elements (`--ring` token provides 2px focus ring)
- [x] Touch targets ≥ 44×44px — not applicable (admin is 768px+ only; no mobile touch targets)
- [x] No time limits, or user can extend them (provisioning has no user-facing timeout; sessions follow standard 24h expiry with refresh)

### Understandable

- [x] Page language declared (`lang="en"` on `<html>`)
- [x] Error messages identify the field and describe the issue (inline below field with `aria-describedby`; per NFR-005)
- [x] All inputs have visible labels (no placeholder-only inputs; per design system anti-pattern §11)
- [x] Required fields indicated by `*` marker and `aria-required="true"` (not by color alone)
- [x] Consistent navigation across pages (AdminSidebarNav identical on all screens per portal)
- [x] Consistent component behavior across pages (same DataTable, Button, Modal patterns everywhere)

### Robust

- [x] Valid HTML structure: headings (`h1` page title, `h2` sections, `h3` subsections), `<nav>` landmarks, `<main>` content area, `<footer>` if present
- [x] All form elements have associated `<label>` elements (via `htmlFor` or wrapping)
- [x] Status updates announced via `aria-live`: Toast notifications (`aria-live="polite"`), provisioning progress (`aria-live="polite"`), health checks (`aria-live="polite"`), capped results warning (`aria-live="assertive"`)
- [x] Modals have focus trap + `role="dialog"` or `role="alertdialog"` + `aria-labelledby` + `aria-describedby`
- [x] Custom components have correct ARIA roles: accordion (`aria-expanded`), switch (`role="switch"`, `aria-checked`), progressbar (`role="progressbar"`, `aria-valuenow`), listbox (`role="listbox"`, `aria-activedescendant`)

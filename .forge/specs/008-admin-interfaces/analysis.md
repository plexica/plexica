## 4. Issues Found

### [HIGH] ISSUE-001: Existing Error Format Violations Not Remediated

**Dimension**: Constitution Compliance (Art. 6.2)  
**Location**: `apps/core-api/src/routes/admin.ts` — lines 640, 655, 1157, 1163, 1358, 1364, 1426, 1439, 1492, 1521, 1527, 1720, 1726, 1774, 1835, 1883, 1949

**Issue**: The plan extends `admin.ts` with 7 new routes (T008-11 through T008-17) but does NOT include a task to fix the 17 existing error responses that use the non-standard format `{ error: 'Not Found', message: '...' }` instead of the constitutionally mandated `{ error: { code: string, message: string } }`.

**Impact**: After implementation, the same route file will contain both compliant and non-compliant error handling. This creates confusion for developers, violates Art. 6.2, and makes the error contract unpredictable for API consumers.

**Recommendation**: Add a task (e.g., T008-00 in Phase 2) to normalize all existing `admin.ts` error responses to Art. 6.2 format before or alongside adding new routes. Estimated effort: 2–3 hours.

---

### [HIGH] ISSUE-002: Edge Case #6 — Spec Constraint Silently Overridden

**Dimension**: Spec-Plan Coherence  
**Location**: Plan §3.1.8, Edge Case #6; Spec §6, Edge Case #6

**Issue**: Spec §6 explicitly states "max 10,000 entries per query" for large audit log date ranges. The plan §3.1.8 explicitly says "No hard cap enforced at query level" and relies on pagination + indexes instead. This is a deliberate deviation from the spec, but it is **not documented as a formal inline decision** in §9 (Architectural Decisions) alongside the PATCH vs PUT decision.

**Impact**: The spec's 10,000-entry cap may have been specified for a reason (e.g., protecting against unbounded COUNT queries, limiting memory usage in the ORM). Silently dropping it without formal documentation means the rationale is lost and cannot be reviewed.

**Recommendation**: Either:

1. Add a formal inline decision in plan §9 documenting why the cap was dropped (recommended), OR
2. Implement the cap as a server-side limit: if `meta.total > 10000`, truncate results and return a `meta.truncated: true` flag.

### [MEDIUM] ISSUE-003: `requireTenantAdmin` Middleware Uses Non-Existent `request.token` Fallback

**Dimension**: Correctness  
**Location**: Plan §4.6, line 976

**Issue**: The plan's `requireTenantAdmin` implementation checks `request.user?.roles ?? request.token?.realm_access?.roles ?? []`. However, in the existing codebase:

- `request.user.roles` is populated by `authMiddleware` (confirmed at auth.ts line 156)
- `request.token` exists (set at auth.ts line 160) but contains the **raw JWT payload** — its structure is `{ iat, exp, sub, ... }`
- The `request.token?.realm_access?.roles` path assumes Keycloak JWT structure, but `extractUserInfo()` already normalizes roles into `request.user.roles`

Meanwhile, the existing `requireRole()` function at auth.ts line 222 only checks `request.user.roles` — no `request.token` fallback. The `requireTenantAdmin` code diverges from this established pattern.

**Impact**: The fallback to `request.token?.realm_access?.roles` is dead code at best, and at worst could produce different behavior than `requireRole()` for the same request. This creates two different authorization code paths.

**Recommendation**: Use the established pattern: `const roles = request.user?.roles ?? []`. Better yet, reuse `requireRole()`:

```typescript
export const requireTenantAdmin = requireRole('tenant_admin', 'tenant_owner', 'admin');
```

This is exactly what `tenant-plugins-v1.ts` does at line 33 (`const requireTenantPluginAdmin = requireRole('tenant_admin', 'tenant_owner', 'admin')`).

### [MEDIUM] ISSUE-004: Missing Rate Limits for Tenant Admin Endpoints

**Dimension**: Security / Performance  
**Location**: Plan §3.2 (all Tenant Admin routes)

**Issue**: The plan specifies rate limits for Super Admin endpoints (60 req/min for dashboard, 30 req/min for audit logs) but does NOT specify any rate limits for Tenant Admin endpoints. The Tenant Admin API includes 15 new endpoints, some of which perform expensive operations (user listing with joins, team member management, audit log queries).

**Impact**: Without rate limits, a compromised or malicious tenant admin account could perform denial-of-service attacks against shared infrastructure (database, Keycloak). The tenant audit log endpoint (§3.2.19) performs the same heavy queries as the super admin audit log endpoint but without the 30 req/min limit.

**Recommendation**: Add rate limits to Tenant Admin endpoints:

- Dashboard: 60 req/min (same as Super Admin)
- Audit logs: 30 req/min (same as Super Admin)
- Mutation endpoints (invite, deactivate, team CRUD): 30 req/min
- Read endpoints (user list, team list, roles): 120 req/min

### [MEDIUM] ISSUE-005: `requireTenantAdmin` Duplicates Existing `requireRole()` Pattern

**Dimension**: Maintainability  
**Location**: Plan §4.6 vs existing `tenant-plugins-v1.ts` line 33

**Issue**: The plan creates a new `requireTenantAdmin` function with custom logic, but the codebase already has an identical capability via `requireRole()`. The existing `tenant-plugins-v1.ts` already uses:

```typescript
const requireTenantPluginAdmin = requireRole('tenant_admin', 'tenant_owner', 'admin');
```

Creating a second, differently-implemented middleware for the same role check introduces unnecessary duplication and divergent authorization paths.

**Impact**: Two different middleware functions checking the same roles with different implementations. Future role changes (e.g., adding a new admin role) would need to be updated in two places.

**Recommendation**: Replace the custom `requireTenantAdmin` implementation with:

```typescript
export const requireTenantAdmin = requireRole('tenant_admin', 'tenant_owner', 'admin');
```

This is a one-liner, reuses the battle-tested `requireRole()`, and follows established patterns.

### [MEDIUM] ISSUE-006: No Migration Script for Existing Tenant Schemas

**Dimension**: Correctness / Operational  
**Location**: Plan §2.4, Migration 003

**Issue**: The plan states: "Existing tenants: run a one-time migration script to add `team_members` to all existing tenant schemas." However, no task in the implementation phases creates this migration script. T008-03 only says "Add `team_members` table to `schema-step.ts` + write migration script for existing tenants" but the task description implies it's part of the same task, while the file map (§5) does not list a migration script file.

**Impact**: If the migration script is forgotten, existing tenants will not have the `team_members` table. Team management endpoints will fail with SQL errors for any tenant provisioned before this change.

**Recommendation**:

1. Add an explicit file to create in §5: `apps/core-api/scripts/migrate-team-members.ts`
2. Add an integration test that verifies the migration script works on an existing tenant schema.
3. Add a rollback script.

### [LOW] ISSUE-008: Spec Uses PUT, Plan Uses PATCH — No Spec Amendment

**Dimension**: Spec-Plan Coherence  
**Location**: Plan §9, Inline Decision: PATCH vs PUT; Spec §8 API tables

**Issue**: The spec's API tables (§8) explicitly specify `PUT` for update endpoints (`PUT /api/v1/admin/tenants/:id`, `PUT /api/v1/tenant/users/:id`, etc.). The plan correctly uses `PATCH` for consistency with the codebase and documents the deviation in §9. However, the spec itself is not updated to reflect this decision, creating a permanent discrepancy between spec and plan.

**Impact**: Future developers reading the spec will expect PUT endpoints. Low severity because the plan clearly documents the deviation.

**Recommendation**: Update spec §8 API tables to use PATCH, or add a note to the spec acknowledging the deviation. This should happen before the spec moves from "Approved" to "Implemented."

**Dimension**: Spec-Plan Coherence  
**Location**: Plan §3.2.15; Spec §8 Tenant Admin APIs table

**Issue**: The spec's Tenant Admin API table (§8) does not include a `DELETE` endpoint for roles. The plan adds `DELETE /api/v1/tenant/roles/:id` (§3.2.15). While this is a logical addition (you need to delete custom roles), it's undocumented scope creep.

### [LOW] ISSUE-010: Audit Log `ip_address` as String — No IPv6 Validation Mentioned

**Dimension**: Correctness  
**Location**: Plan §2.1, `core.audit_logs` table

**Issue**: The plan uses `String?` for `ip_address` instead of PostgreSQL `INET` (due to Prisma limitation) and mentions "validated via Zod before insertion." However, the plan does not specify what the Zod validation schema looks like. IP addresses can be IPv4, IPv6, or `::ffff:` mapped — the Zod schema needs to handle all formats.

**Impact**: Low — if Zod validation is too strict (e.g., only validates IPv4), legitimate IPv6 addresses from clients could be silently dropped.

**Recommendation**: Document the Zod validation schema explicitly. Recommend `z.string().ip()` which handles both IPv4 and IPv6 in modern Zod versions.

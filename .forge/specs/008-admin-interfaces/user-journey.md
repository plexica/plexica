## 2. Journey Maps

### Journey 1: Super Admin Creates a New Tenant

**Persona**: Marcus (Super Admin)
**Trigger**: Sales team emails Marcus requesting a new tenant for "Acme Corp" with admin email `admin@acme.com`.
**Goal**: Provision a fully functional tenant in under 60 seconds.
**FR coverage**: FR-001, FR-002
**NFR coverage**: NFR-005 (form validation), NFR-008 (provisioning feedback)
**Edge cases**: #1 (duplicate slug), Edge Case #3 (referenced for context on delete flow)
**Screens**: 1 (Dashboard), 2 (Tenant List), 3 (Tenant Create Wizard)

#### Happy Path

| Step | Action                                        | Screen               | System Response                                                                                                         | Emotion      |
| ---- | --------------------------------------------- | -------------------- | ----------------------------------------------------------------------------------------------------------------------- | ------------ |
| 1    | Marcus navigates to `/super-admin`            | SA Dashboard (Scr 1) | Dashboard loads with 4 StatCards (42 tenants, 1250 users, 12 plugins, ✅ Health) and recent activity table              | Neutral      |
| 2    | Clicks "Tenants" in sidebar                   | SA Tenant List (2)   | Tenant list loads: 42 rows, sorted by name, paginated. Search bar and status filter available                           | Neutral      |
| 3    | Clicks "+ New Tenant" button                  | SA Wizard Step 1 (3) | Wizard opens on Step 1: Details. Step indicator shows ● Details, ○ Configure, ○ Provisioning. Name field has focus      | Focused      |
| 4    | Types "Acme Corp" in Name field               | SA Wizard Step 1 (3) | Slug auto-generates: `acme-corp`. Slug field updates in real-time as Marcus types                                       | Efficient    |
| 5    | Tabs to Admin Email; types `admin@acme.com`   | SA Wizard Step 1 (3) | Email validated on blur: green checkmark. "Next: Configure →" button enables                                            | Confident    |
| 6    | Clicks "Next: Configure →"                    | SA Wizard Step 2 (3) | Step 2: Configure loads. Theme colors default to platform defaults. Plugin checkboxes listed. Max users field shows 500 | Neutral      |

| Step | Action                                              | Screen               | System Response                                                                                                       | Emotion    |

#### Edge Case Flow B: Provisioning Failure (Edge Case #5 — adapted for tenant)

**Recovery**: Automatic rollback means no orphaned resources. Retry is one click. Marcus doesn't need to re-enter form data.

**NFR coverage**: NFR-005 (form validation)

**Screens**: 11 (Dashboard), 12 (User List), 13 (Invite Modal), 16 (Role Editor — if role doesn't exist yet)

| 1    | Priya navigates to `/admin`                  | TA Dashboard (11) | Dashboard loads: 45 users (42 active, 3 invited), 6 teams, 3 plugins. Recent activity shows latest audit entries    | Oriented     |
| 2    | Clicks "Users" in sidebar                    | TA User List (12) | User list loads: 45 rows with Name, Email, Role badge, Status badge, Actions column. Search and role/status filters | Neutral      |
| 3    | Clicks "+ Invite User" button                | Invite Modal (13) | Modal opens with focus on Email input. Fields: Email (required), Role dropdown (required, defaults to "User")       | Ready        |

| Step | Action                                   | Screen            | System Response                                                                                                                                             | Emotion   |
| ---- | ---------------------------------------- | ----------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- | --------- |

#### Edge Case Flow B: Last Tenant Admin Prevention (Edge Case #7)

| Step | Action                                                           | Screen            | System Response                                                                                                                                                            | Emotion    |
| ---- | ---------------------------------------------------------------- | ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------- |
| 7a   | Priya clicks ⋮ → "Change Role" on Sara Lee (last `tenant_admin`) | TA User List (12) | Change Role dialog opens: Current role: `tenant_admin`. Dropdown shows other roles                                                                                         | Neutral    |
| 7b   | Selects "User" from dropdown; clicks "Save"                      | TA User List (12) | API returns 403. Prevention dialog appears: ❌ "Cannot change role. Sara Lee is the last user with the tenant_admin role. Assign tenant_admin to another user first." [OK] | Stopped    |
| 7c   | Clicks "OK, Understood"                                          | TA User List (12) | Dialog closes. Focus returns to Sara Lee's ⋮ button. No changes made. Priya understands she needs to promote someone else first                                            | Redirected |

**Recovery**: The system prevents the dangerous operation _before_ any data changes. The error message is specific and actionable — it tells Priya exactly what to do (assign `tenant_admin` to another user first).

#### Edge Case Flow A: System Role Read-Only (Edge Case #2)

| Step | Action                                       | Screen              | System Response                                                                                                                                                    | Emotion     |
| ---- | -------------------------------------------- | ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------- |
| 2a   | Daniel clicks "🔒 tenant_admin" in role list | TA Role Editor (16) | Right panel loads tenant_admin details. 🔒 Info banner: "This is a system role. System roles cannot be modified or deleted." All fields are read-only (grayed out) | Informed    |
| 2b   | Daniel tries to click a permission checkbox  | TA Role Editor (16) | Checkbox doesn't respond (`aria-disabled="true"`). Tooltip on hover: "System role — cannot be edited." No Delete or Save buttons visible                           | Understands |

| ---- | ------------------------------------------ | ------------------- | ---------------------------------------------------------------------------------------------------------------- | ---------- |

| 11b  | Daniel changes name to "Sales Rep"         | TA Role Editor (16) | Error clears on edit. "Save Role" re-enables. Daniel saves successfully                                          | Relieved   |

**Recovery**: Inline error is specific and actionable. Form state (selected permissions) is preserved — Daniel only needs to change the name.

## 3. Emotional Journey Summary

### Marcus (Super Admin) — Tenant Creation Flow

```
Neutral → Focused → Efficient → Anticipation → Engaged → Satisfied → Accomplished
    │                                 │
    └── Edge: Frustrated → Relieved   └── Edge: Concerned → Determined → Relieved
        (duplicate slug)                   (provisioning failure)
```

**Delight points**: Real-time provisioning progress with per-step feedback; slug auto-generation saves typing; one-click retry preserves form data.

**Frustration points**: Duplicate slug (mitigated by immediate inline error + easy edit); provisioning failure (mitigated by automatic rollback + one-click retry).

---

### Priya (Tenant Admin) — User Invite + Role Assignment

```
Oriented → Neutral → Ready → Confident → Satisfied → Accomplished
    │                    │
    └── Edge: Informed   └── Edge: Stopped → Redirected
        (existing user)       (last admin prevention)
```

**Delight points**: Automatic existing-user detection with clear info banner and CTA change; role dropdown includes custom roles immediately after creation.

**Frustration points**: Last-admin prevention blocks the action (mitigated by clear explanation and specific remediation: "assign tenant_admin to another user first").

---

### Daniel (New Tenant Admin) — Custom Role Creation

```
Curious → Exploring → Focused → Learning → Methodical → Confident → Accomplished → Satisfied
    │          │
    └── Edge:  └── Edge: Informed → Understands → Redirected
    (dup name)      (system role read-only)
```

**Delight points**: Permission groups organized by source (Core vs. CRM vs. Billing) make it easy to find relevant permissions; wildcard checkbox concept (select all sub-permissions) reduces clicking; permission count in accordion header ("CRM Plugin 2/7") gives progress visibility.

**Frustration points**: Unfamiliar permission names (mitigated by grouping by plugin source); system role appearing editable then being disabled (mitigated by 🔒 icon in list + info banner + disabled state being obvious).

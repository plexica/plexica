## 2. User Journeys

---

### Journey 1: Sofia — Switching to Italian

**Trigger**: Sofia logs into Plexica and sees the UI in English. She wants to switch to Italian.

**Preconditions**:

- Sofia is logged into her tenant's Plexica instance
- Italian (it) is a supported locale for at least the core namespace
- The CRM plugin has Italian translations available

#### Happy Path

| Step | User Action                                                         | System Response                                                                                                                                    | Emotional State                                  |
| ---- | ------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------ |
| 1    | Notices Globe icon (`Globe`) in the app header or user profile area | Language selector button is visible with current locale label ("English")                                                                          | 😐 Neutral — looking for language options        |
| 2    | Clicks the Globe icon / language selector                           | Dropdown opens showing languages in native script: "English", "Italiano", "Español", "Deutsch"                                                     | 😊 Relieved — clear, recognizable language names |
| 3    | Selects "Italiano"                                                  | Dropdown closes. All UI text re-renders in Italian without page reload (React context re-render, per FR-008). Toast: "Lingua cambiata in Italiano" | 🎉 Satisfied — instant switch, no page loss      |
| 4    | Navigates to CRM Contacts page                                      | CRM plugin text also renders in Italian (namespace `crm` loaded for `it` locale, per FR-005)                                                       | 🎉 Satisfied — everything is consistent          |
| 5    | (Background) Preference saved to `User.locale`                      | On next login, Italian loads automatically (per FR-009 detection chain)                                                                            | 😊 Delighted — remembered preference             |

**Outcome**: Sofia uses the platform entirely in Italian. Her preference persists across sessions.

**Delight opportunities**:

- Step 2: Show language names in their native script so Sofia can immediately recognize "Italiano" without reading English labels
- Step 3: Instant re-render with no spinner or page reload gives a feeling of responsiveness
- Step 5: Automatic persistence means she never has to switch again

#### Edge Case A: Locale Partially Supported

**Trigger**: Sofia's browser locale is `it-CH` (Italian, Switzerland). The platform supports `it` but not `it-CH`.

| Step | User Action                                                  | System Response                                                                                          | Emotional State                                           |
| ---- | ------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------- | --------------------------------------------------------- |
| 1    | Opens Plexica for the first time                             | Locale detection chain (FR-009): `it-CH` (browser) → no exact match → falls back to `it` → loads Italian | 😐 Neutral — may not notice                               |
| 2    | Notices some CRM keys display in English                     | CRM plugin only supports `en` and `de` — Italian fallback for CRM falls to `en` (per FR-003)             | 🤔 Slightly confused — mixed languages                    |
| 3    | Opens language selector, sees "Italiano" is already selected | Dropdown shows "Italiano" as active. No error message                                                    | 😐 Accepts the situation — understands CRM has no Italian |

**Recovery path**: No recovery needed — fallback behavior is by design. Sofia sees core UI in Italian and CRM in English. If CRM adds Italian later, it automatically resolves.

**Feedback**: No explicit error. The system degrades gracefully per FR-003.

#### Edge Case B: Translation File Fails to Load (CDN Error)

**Trigger**: Sofia switches to Italian, but the CDN serving `it/core.json` is temporarily unavailable (Edge Case #3 in spec).

| Step | User Action                               | System Response                                                                                  | Emotional State                               |
| ---- | ----------------------------------------- | ------------------------------------------------------------------------------------------------ | --------------------------------------------- |
| 1    | Selects "Italiano" from language selector | Network request for `it/core.json` fails (CDN 503)                                               | —                                             |
| 2    | —                                         | System falls back to cached `en` translations. UI remains in English. Warning logged server-side | 😤 Frustrated — expected Italian, got English |
| 3    | Sees UI still in English                  | Toast warning: "Unable to load Italian translations. Displaying in English."                     | 😕 Confused but informed                      |
| 4    | Tries again in 30 seconds                 | Retry succeeds. UI re-renders in Italian                                                         | 😊 Relieved                                   |

**Recovery path**: Automatic retry on next navigation or manual retry via language selector.

**Feedback**: Warning toast: "Unable to load Italian translations. Displaying in English."

**Recovery path**: Not needed — cancellation is automatic. User preference saved as `es`.

### Journey 2: Marco — Overriding "Deals" to "Opportunities"

**Trigger**: Marco's sales team uses the term "Opportunities" internally. The CRM plugin uses "Deals". Marco wants to customize the CRM terminology for his tenant.

**Preconditions**:

- Marco is logged in with `tenant_admin` role
- CRM plugin is enabled for the tenant
- CRM plugin has `en` translations with the key `crm:deals.title` = "Deals"

#### Happy Path

| Step | User Action                                       | System Response                                                                                                                       | Emotional State                              |
| ---- | ------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| 1    | Navigates to Tenant Admin → Translation Overrides | Override Editor page loads. Shows searchable list of all translation keys across enabled plugins                                      | 😐 Neutral — looking for the right key       |
| 2    | Types "deals" in the search field                 | Table filters to show keys matching "deals": `crm:deals.title`, `crm:deals.list.empty`, `crm:deals.new`                               | 😊 Relieved — found it quickly               |
| 3    | Clicks on `crm:deals.title` row                   | Row expands or inline editor opens. Shows: **Original**: "Deals" (from CRM plugin). **Override**: (empty)                             | 😐 Neutral — understands the UI              |
| 4    | Types "Opportunities" in the Override input field | Preview panel live-updates: shows "Deals" → "Opportunities" in context                                                                | 😊 Delighted — sees the change before saving |
| 5    | Clicks "Save Overrides" button                    | API call `PUT /api/v1/tenant/translations/overrides`. Success toast: "Translation overrides saved." Override badge appears on the row | 🎉 Satisfied — change is saved               |
| 6    | Navigates to CRM Deals page                       | Page title shows "Opportunities" instead of "Deals" (per FR-007)                                                                      | 🎉 Satisfied — it works everywhere           |

**Outcome**: The CRM "Deals" label is replaced with "Opportunities" across the tenant's Plexica instance. The override persists through plugin updates (per US-003 AC).

**Delight opportunities**:

- Step 2: Instant search filtering with namespace grouping makes finding keys fast
- Step 4: Live preview eliminates the save-check-reload cycle
- Step 5: Success toast confirms the change was applied immediately

#### Edge Case B: Orphaned Override Key (Plugin Update Removes Key)

**Trigger**: CRM plugin v2.0 renames `crm:deals.title` to `crm:opportunities.title`. Marco's override for the old key becomes orphaned (Edge Case #4, FR-014).

| Step | User Action                                   | System Response                                                                                               | Emotional State                           |
| ---- | --------------------------------------------- | ------------------------------------------------------------------------------------------------------------- | ----------------------------------------- |
| 1    | Opens Override Editor after CRM plugin update | Override list loads. Row for `crm:deals.title` shows a warning badge: `AlertTriangle` icon + "Orphaned" badge | 🤔 Uncertain — what does "orphaned" mean? |
| 2    | Hovers over the warning badge                 | Tooltip: "This key no longer exists in the CRM plugin. The override has no effect. You can safely remove it." | 😐 Informed — understands the situation   |
| 3    | Clicks "Remove" on the orphaned row           | Confirmation dialog: "Remove override for `crm:deals.title`? This key is no longer used by the CRM plugin."   | 😐 Neutral                                |
| 4    | Confirms removal                              | Override deleted. Toast: "Override removed." Row disappears from list                                         | 😊 Relieved — cleaned up                  |

**Recovery path**: Orphaned overrides are harmless — they have no effect. Marco can remove them at his convenience.

**Feedback**: Warning badge with tooltip explanation. No blocking error.

#### Edge Case C: Override Payload Exceeds Size Limit

**Trigger**: Marco (or an automated import) attempts to save overrides that exceed the 1MB payload limit.

| Step | User Action                                          | System Response                                                                                                                                        | Emotional State            |
| ---- | ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------- |
| 1    | Pastes a very large number of overrides (>1MB total) | Client-side: character count warning appears when approaching limit                                                                                    | 😐 Neutral                 |
| 2    | Clicks "Save Overrides"                              | API returns 413 `PAYLOAD_TOO_LARGE`. Error toast: "Override payload exceeds the 1MB limit. Reduce the number of overrides or shorten override values." | 😤 Frustrated — can't save |
| 3    | Removes some overrides                               | Payload size indicator updates. Below limit                                                                                                            | 😐 Trying again            |
| 4    | Clicks "Save Overrides" again                        | Success. Toast: "Translation overrides saved."                                                                                                         | 😊 Relieved                |

**Feedback**: "Override payload exceeds the 1MB limit. Reduce the number of overrides or shorten override values."

### Journey 3: Yuki — Dev-Mode Missing Key Detection

**Trigger**: Yuki is developing the billing plugin locally and wants to verify all translation keys render correctly. She's running Plexica in development mode.

**Preconditions**:

- Plexica is running in `NODE_ENV=development`
- Billing plugin is enabled with namespace `billing`
- Yuki has added new UI text but hasn't yet created all translation keys

#### Happy Path

| Step | User Action                                             | System Response                                                                                                                                                                  | Emotional State                              |
| ---- | ------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------- |
| 1    | Navigates to the billing plugin's invoice page          | Page renders. Most text appears correctly. Two labels show raw key paths: `billing:invoice.tax_label`, `billing:invoice.discount_label` — each wrapped in a red border highlight | 😐 Neutral — expected some missing keys      |
| 2    | Inspects the highlighted elements                       | Red dashed border + tooltip: "Missing translation: `billing:invoice.tax_label` (locale: en)"                                                                                     | 😊 Helpful — knows exactly which keys to add |
| 3    | Adds the missing keys to `translations/en/billing.json` | Hot-reload picks up the change. Red borders disappear. Text renders correctly                                                                                                    | 🎉 Satisfied — fast feedback loop            |
| 4    | Switches to Italian locale to check `it` translations   | New missing keys appear with red borders for Italian-only gaps. English fallback text also shown in smaller font below                                                           | 😐 Neutral — knows what to do                |

**Outcome**: Yuki identifies and fixes all missing translation keys before shipping the plugin. No missing keys reach production.

**Delight opportunities**:

- Step 2: Tooltip with the exact key path means Yuki can copy-paste directly into her JSON file
- Step 3: Hot-reload integration means no manual refresh needed

#### Edge Case A: Invalid Key Format in Plugin Manifest

**Trigger**: Yuki's billing plugin manifest includes a translation key that violates the validation rules (FR-011): `billing:invoice.tax label` (contains a space).

| Step | User Action                                                     | System Response                                                                                                                                                | Emotional State                     |
| ---- | --------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------- |
| 1    | Attempts to register the billing plugin via the Plugin Registry | Registration rejected with 400 error                                                                                                                           | 😤 Frustrated — registration failed |
| 2    | Reads the error response                                        | Error: "Invalid translation key: `invoice.tax label`. Keys must contain only alphanumeric characters, dots, and underscores. Maximum 128 characters." (FR-011) | 😐 Informed — clear fix path        |
| 3    | Renames key to `invoice.tax_label` and re-registers             | Registration succeeds                                                                                                                                          | 😊 Relieved                         |

**Feedback**: "Invalid translation key: `invoice.tax label`. Keys must contain only alphanumeric characters, dots, and underscores. Maximum 128 characters."

---

#### Edge Case B: Translation File Too Large

**Trigger**: Yuki's billing plugin has a single namespace file `billing.json` that exceeds 200KB (FR-012, Edge Case #5).

| Step | User Action                                                                                     | System Response                                                                                        | Emotional State                   |
| ---- | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ | --------------------------------- |
| 1    | Attempts to register the billing plugin                                                         | Registration rejected with error                                                                       | 😤 Frustrated                     |
| 2    | Reads the error response                                                                        | Error: "Translation file for namespace 'billing' exceeds 200KB limit. Split into multiple namespaces." | 😐 Informed — actionable guidance |
| 3    | Splits `billing.json` into `billing-invoices.json` and `billing-reports.json`, updates manifest | Registration succeeds                                                                                  | 😊 Relieved                       |

**Feedback**: "Translation file for namespace 'billing' exceeds 200KB limit. Split into multiple namespaces."

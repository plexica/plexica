# Design Spec — Spec 005: Super Admin Panel

> Generated by `/forge-ux`. Covers FR IDs 005-01…005-11 from `.forge/specs/005-super-admin/spec.md`.
> **Platform**: Desktop-only internal tool (1440px target, 1280px minimum). Mobile-responsive is NOT required — this is explicitly documented as a scoped decision: the admin panel is used by operators on workstations, never on mobile devices. Dark mode supported via existing `[data-theme="dark"]` tokens.
> **Stack**: React 19, TanStack Router/Query, Zustand, Tailwind, Radix UI, react-intl, Lucide icons. All components reference `@plexica/ui` where available.

## Information Architecture

```
Admin App (master realm auth)
├── /login                     ← Keycloak password grant
├── /dashboard                 ← KPI cards + status breakdown (005-01)
├── /tenants                   ← Searchable, filterable table (005-02)
│   └── /tenants/$tenantId     ← Detail with tabs (005-03)
│       ├── Info tab           ← Tenant metadata + lifecycle actions
│       │   ├── [Suspend Dialog]    ← 005-05
│       │   ├── [Reactivate Dialog] ← 005-06
│       │   ├── [Delete Dialog]     ← 005-07 (type-to-confirm)
│       │   └── [Deletion Status]   ← 005-07 (conditional, when pending_deletion)
│       ├── Users tab          ← User count (read-only aggregate)
│       ├── Plugins tab        ← Installed plugins (read-only)
│       └── Audit tab          ← Platform audit entries for this tenant
├── /provision                 ← 3-step wizard (005-04)
├── /plugins                   ← Catalog table + review queue (005-08)
├── /health                    ← Service status grid (005-09)
├── /logs                      ← Log viewer with filters (005-10)
└── /kafka                     ← Consumer lag table + DLQ summary (005-11)
```

**Navigation pattern**: List → Detail (drill-down). Tabs within detail for multi-facet views. Suspend/Reactivate/Delete are modal dialogs (not routes) — keeps the user in context. URL state preserves filters on all list views (data-presentation §2.3: context preservation).

**Entry/exit points**: Sidebar (7 nav items, already implemented in `sidebar.tsx`) is the primary navigation. Dashboard KPI cards link to Tenants/Health. Health Kafka card links to Kafka Status. Kafka page links to Logs. Logs entries link to Tenant Detail. All cross-links preserve filter context via URL search params.

  - Empty (filtered): "No tenants match 'acm'. [Clear search]"

  - Loading: 5 skeleton rows (Table component with pulse animation)
  - Error: "Failed to load tenants. [Retry]" above table
  - Partial: N/A (single query, no partial failure)

  Justification (data-presentation §4.1): casual-to-power users, ≤3 dimensions,
  finite status values → inline chips + dropdown, not sidebar or query builder.

Accessibility: Table with scope="col" headers (existing Table component);
  search input has label (sr-only) + aria-label; status filter is Select with
  aria-label; pagination has aria-labels on buttons; aria-live="polite" on

### New Admin-Specific Token

One new semantic color token for `pending_deletion` status (distinct from existing warning/amber to convey higher severity — "imminent irreversible deletion"):

```css
/* Add to packages/ui/src/tokens/colors.css */
:root {
  --color-status-pending-deletion:       #ea580c; /* orange-600 */
  --color-status-pending-deletion-light: #fed7aa; /* orange-200 */
  --color-status-pending-deletion-dark:  #9a3412; /* orange-800 */
}

[data-theme="dark"] {
  --color-status-pending-deletion:       #f97316; /* orange-500 — brighter in dark */
  --color-status-pending-deletion-light: #7c2d12; /* orange-900 */
  --color-status-pending-deletion-dark:  #fed7aa; /* orange-200 */
}
```

**Rationale**: The 4 tenant statuses need distinct, non-overlapping colors. `active` = green (success), `suspended` = amber (warning — temporary, reversible), `pending_deletion` = orange (higher severity than warning, irreversible in progress), `deleted` = neutral gray (terminal). Orange sits between amber and red on the severity spectrum, correctly signaling "deletion in progress, more urgent than suspension but not yet failed."

### Badge Variant Mapping (TenantStatusBadge)

The existing `@plexica/ui` Badge component has variants: `admin`, `member`, `viewer`, `pending`, `success`, `error`, `default`. Tenant status badges map to these + the new token:

| Tenant Status | Badge Implementation | Color Source |
|---------------|----------------------|-------------|
| active | `variant="success"` + `CheckCircle2` icon | `--color-success` |
| suspended | `variant="pending"` + `PauseCircle` icon | `--color-warning` |
| pending_deletion | custom className using new token + `AlertTriangle` icon | `--color-status-pending-deletion` |
| deleted | `variant="default"` + `XCircle` icon | `--color-neutral-400` |

> **Note**: The `pending_deletion` badge requires either a new Badge variant or a custom className override. Recommendation: add a `warning` variant to Badge (using the new orange token) to avoid inline styles. This is a minor design system extension, not an ADR-level change.

## Accessibility (WCAG 2.1 AA) — Per Screen

| Screen | Key A11y Requirements |
|--------|-----------------------|
| All screens | Skip-nav link (first focusable element); logical tab order (sidebar → content); focus indicator visible (`--focus-ring-width: 2px`, `--color-primary-500`); all interactive elements keyboard-accessible |
| Login | Form labels visible; error messages in text (not color); `aria-live` on error; password show/hide toggle |
| Dashboard | KPIs in `<dl>`; `aria-live="polite"` on refresh timestamp; cards are static (no interactive without keyboard equivalent) |
| Tenant List | Table `scope="col"` + `aria-sort`; search `aria-label`; filter Select `aria-label`; pagination `aria-label`s; result count `aria-live` |
| Tenant Detail | Tabs use @plexica/ui Tabs (Radix Tabs — keyboard arrow nav, `role="tablist"`); tab state in URL (`?tab=info`); `aria-labelledby` for tab panels |
| Provisioning Wizard | Stepper as `<ol>` + `aria-current="step"`; form labels visible; validation on blur; temp password Copy button announces via `aria-live` |
| Suspend/Reactivate Dialog | Radix Dialog (focus trap + Esc); `role="dialog"` + `aria-labelledby`; confirm button has descriptive `aria-label` |
| Delete Dialog | TypeToConfirmDialog (existing a11y: focus to input, role="alert" on error); input `aria-label` = instructions |
| Deletion Status | `aria-live="polite"` on step updates; status icons have sr-only text; retry button `aria-label` includes step name |
| Plugin Catalog | Table a11y (same as Tenant List); review dialog focus trap; notes textarea visible label |
| Health Dashboard | `<figure>` + `<figcaption>` per card; status icon + text (not color alone); degraded/down `role="alert"`; aggregate `aria-live` |
| Logs Viewer | Table a11y; level badges (icon + text); expandable rows `aria-expanded`; filter `aria-label`s; result count `aria-live`; no PII (Pino redact) |
| Kafka Status | Tables a11y; lag bar with sr-only unit; warning badges (icon + text); `aria-label` includes threshold context |

### Cross-Cutting A11y Decisions

1. **Status badges never use color alone** — every badge has icon (Lucide) + text label (WCAG 1.4.1).
2. **Dialogs use Radix UI Dialog** — native focus trap, Esc to close, `aria-modal="true"`, `aria-labelledby` (WCAG 2.1.1, 2.4.3, 4.1.2).
3. **Loading states use skeletons** matching final layout (not spinner-only) — prevents layout shift, perceived performance (WCAG 1.3.2).
4. **All status updates announced** via `aria-live="polite"` (refresh timestamps, saga progress, result counts) — WCAG 4.1.3.
5. **Tables use semantic HTML** — `@plexica/ui` Table component already has `scope="col"`, `aria-sort` (WCAG 1.3.1).
6. **Dark mode** — all tokens have `[data-theme="dark"]` overrides; contrast ratios verified for both modes (WCAG 1.4.3).
7. **No `window.confirm()` or `window.alert()`** — all confirmations use Dialog components (Constitution §frontend patterns).
8. **Skip-nav link** — present on all authenticated screens, first in tab order, jumps to `#main-content`.

## Key UX Decisions

| Decision | Rationale | FR Ref |
|----------|-----------|--------|
| **3-step wizard for provisioning** (not single form) | Creates 3 infra resources (schema, realm, bucket) — wizard separates input → review → execution, prevents errors on irreversible infra creation | 005-04 |
| **Type-to-confirm for deletion** (type slug, not "CONFIRM") | GDPR deletion is irreversible and creates 3 resources to erase. Typing the specific slug prevents wrong-tenant deletion. Existing `TypeToConfirmDialog` component reused. | 005-07 |
| **Status color coding: green/amber/orange/gray** | 4 distinct severity levels. Orange for `pending_deletion` (new token) signals "irreversible in progress" — more urgent than amber (reversible suspension), less common than gray (terminal). | 005-02/05/07 |
| **Deletion saga as async with status panel** (not sync wait) | Deletion takes < 30s but can fail partially. Async saga + polling status panel lets Dana monitor progress and retry individual failed steps without blocking the UI. | 005-07 |
| **Suspend/Reactivate as dialogs** (not separate routes) | Quick reversible actions — dialogs keep Dana in the tenant detail context. Optimistic lock version included in mutation body. | 005-05/06 |
| **Desktop-only** (no mobile-responsive) | Internal tool for operators on workstations. Tables, multi-column dashboards, and dense data views are the primary content. Mobile would degrade the experience without serving any user. Documented as explicit scope. | all |
| **Inline filters for lists** (not sidebar facets) | Tenant list has 2 filterable dimensions (search + status). Plugin catalog has 2 (search + status/reviewStatus). ≤3 dimensions → inline chips per data-presentation §4.1. No query builder needed. | 005-02, 005-08 |
| **Health as status cards** (not charts) | 5 fixed services, monitoring intent (not trend). Cards give visual prominence with icon + color. Aggregate header provides the headline. | 005-09 |
| **Logs via Loki proxy** (not direct query) | Backend translates filters to LogQL. Frontend never knows Loki URL. 503 on Loki down with retry. No PII (Pino redact). | 005-10 |
| **Role-gated UI** (Sam sees no destructive actions) | Read-only super_admin role. Provision/Suspend/Delete/Approve/Publish buttons hidden by role check, not just disabled. Defense-in-depth. | all |

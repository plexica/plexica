# Wireframes — Spec 005: Super Admin Panel

> Generated by `/forge-wireframe`. Covers all 13 screens from the design-spec.md
> screen inventory. Each screen is rendered with **4 states** (default, loading,
> empty, error) plus any screen-specific variants (dialog states, saga failure,
> partial failure).
>
> **Platform**: Desktop-only internal tool (1440px target, 1280px minimum).
> Mobile-responsive is explicitly NOT in scope (documented in design-spec.md).
> Dark mode supported via `[data-theme="dark"]` tokens — wireframes show light
> mode; tokens invert in dark mode.
>
> **Stack**: React 19, TanStack Router/Query, Zustand, Tailwind, Radix UI,
> react-intl, Lucide icons. All components reference `@plexica/ui` where
> available.
>
> **Convention**: `[...]` = annotation; `[FR 005-NN]` = functional requirement
> reference; `[aria-*]` = accessibility note; `[i18n: key]` = i18n string key;
> `[Component]` = `@plexica/ui` component reference.
>
> **Legend**: `✓` = done/success · `⏳` = in progress · `○` = pending · `✗` =
> failed/error · `⚠` = warning/degraded · `▾` = dropdown caret · `▸` = active
> tab · `▒` = skeleton placeholder · `⟳` = spinner / refresh.

## Screen 1: Sign-in Redirect and Callback

**Routes**: `/login`, `/callback` · **FR ref**: auth · **Components**:
`LoginPage`, `AuthCallbackPage`, `SessionExpiredHandler`

Plexica renders no credential fields. `/login` starts a single-flight OIDC
Authorization Code + PKCE S256 redirect. Keycloak owns password, MFA,
WebAuthn, and federated-login UX.

### Redirecting (`/login`)

```
┌──────────────────────────────────────────────────────────────────────────┐
│                      ┌──────────────────────────────────┐                │
│                      │       [ShieldCheck logo]         │                │
│                      │       Plexica Admin              │                │
│                      │  ⟳ Redirecting to secure sign-in │ [aria-live]    │
│                      │  [ Continue to Keycloak ]        │                │
│                      └──────────────────────────────────┘                │
└──────────────────────────────────────────────────────────────────────────┘
```

Automatic redirect and fallback use one guarded action, so repeated effects or
clicks cannot replace in-flight `state`/verifier data. The app sends S256 and
Keycloak enforces `pkce.code.challenge.method=S256` (ADR-023).

### Callback (`/callback`)

```
│  ⟳ Completing secure sign-in…                            [aria-live]     │
│  Verifying the response.                                                 │
```

Validate OAuth parameters and exact anti-CSRF state, exchange the code with
its one-time verifier, remove PKCE data, and navigate to `/dashboard`. Tokens
are in `sessionStorage`; retain the ID token for RP-Initiated Logout.

### Callback / Redirect Error

```
│  Sign-in could not be completed.                         [role="alert"]  │
│  Your secure sign-in response was invalid or expired.                    │
│  [ Try secure sign-in again ]                                            │
```

Missing/replayed state, denial, failed exchange, and Keycloak unavailability
use localized generic messages. Retry clears stale one-time data and starts a
fresh flow. Never display raw OAuth details or tokens.

### Session Expired

```
│  Your session has expired. Redirecting to secure sign-in… [status]       │
```

After refresh failure, withhold protected content and clear all tokens. Keep
the localized notice long enough to announce, then start fresh PKCE via
`/login`; never replace it with an inline password form.

### Accessibility

| Criterion | Implementation |
|-----------|----------------|
| 2.1.1 Keyboard | Fallback/retry buttons are keyboard operable and focus-visible |
| 2.4.3 Focus Order | Error focus moves to alert heading, then retry |
| 3.3.1 Error Identification | Localized text + icon, never color alone |
| 4.1.3 Status Messages | Redirect, callback, and expiry use `aria-live` |

**i18n keys**: `admin.login.redirecting`, `admin.login.continue`,
`admin.callback.loading`, `admin.callback.error`, `admin.callback.retry`,
`admin.session.expired`, `admin.session.redirecting`.

### Key Layout Decisions

1. **Sidebar navigation (7 items)** — Dashboard, Tenants, Provision, Plugins,
   Health, Logs, Kafka. Provision is role-gated (hidden from read-only
   admins). Active item marked with `aria-current="page"` + primary color
   background. Collapsible to icon-only at 1280px.

2. **Dashboard: KPI cards in a row** (4 cards) + status breakdown as labeled
   chips + quick-action cards. KPIs use `<dl>` (definition list) — not
   charts. 4 leading metrics need exact values + simple delta; status
   breakdown is 4 categories needing exact counts (chips > pie chart for
   accessibility and scannability).

3. **Tenant List: inline filter bar** (search + status dropdown) above a
   paginated `AdminDataTable`. Inline filters (not sidebar facets) because
   there are only 2 filterable dimensions (data-presentation §4.1). URL
   state preserves filters across navigation. Row click → detail (no
   selection model).

4. **Tenant Detail: 4 Radix Tabs** (Info / Users / Plugins / Audit) with tab
   state in URL (`?tab=info`). Tabs use `role="tablist"` + arrow-key
   navigation (Radix). Lifecycle action buttons (Suspend/Reactivate/Delete)
   are separate buttons below metadata — more visible than burying in a
   dropdown. Contextual: Suspend disabled when already suspended;
   Reactivate enabled when suspended; Delete always available.

5. **Provisioning: 3-step wizard** with `StepProgressIndicator` (horizontal
   stepper). Step 1 collects input with real-time validation (slug
   uniqueness on blur). Step 2 is a static review (no loading). Step 3
   shows progress + result or conflict. Conflict resolution offers
   "Adopt existing" / "Force cleanup & retry" / "Cancel" (spec §7 edge
   cases for orphaned resources).

6. **Suspend/Reactivate/Delete: modal dialogs** (not separate routes) — keep
   the admin in the tenant detail context. Suspend is warning-variant
   (amber, reversible); Delete is destructive-variant (red, irreversible)
   with type-to-confirm. Reactivate is symmetric to Suspend with
   success-variant confirm. All use Radix Dialog (focus trap + Esc).

7. **Deletion Status Panel: vertical step list** (not horizontal stepper) —
   shows 3 saga steps with status icons, attempts, last error, and per-step
   retry. Auto-refreshes every 5s while pending/in_progress; pauses on
   failure. Replaces the lifecycle action buttons in the Info tab when
   `status === 'pending_deletion'`.

8. **Plugin Catalog: inline filter bar + table** (same pattern as Tenant
   List) + filter chips for review queue (Pending Review count in chip
   label). Review dialog shows full manifest summary + notes textarea.
   Unpublish warning dialog lists affected tenants (deprecate if
   installed, unpublished if not).

9. **Health Dashboard: 3+2 card grid** (5 fixed services). Cards are
   `<figure>` with `<figcaption>` (service name) + `<dl>` (status,
   latency). Kafka card is the only interactive one (navigates to
   `/kafka`). Aggregate header provides the headline ("3 of 5 healthy")
   before the detail. Down services use `role="alert"` + `aria-live=
   "assertive"` (critical announcements interrupt screen readers).

10. **System Logs: inline filter bar + table** with explicit "Search"
    button (Loki queries are expensive — no auto-search on filter change).
    Load-more pagination (not page numbers — Loki returns newest first).
    Expandable rows show full metadata JSON in `<pre>`. All PII redacted
    by Pino redact paths. Two distinct error states: Loki-down (503
    SERVICE_UNAVAILABLE) vs query-timeout (503 LOG_QUERY_TIMEOUT) with
    different recovery suggestions.

### Cross-Cutting Patterns

- **Skeletons, not spinners** — every loading state uses skeleton
  placeholders matching the final layout (prevents layout shift, WCAG 1.3.2).
  Spinners only appear on buttons during mutations (submit, retry).
- **Role-gated UI** — Provision, Suspend, Reactivate, Delete, Approve,
  Reject, Publish, Unpublish, Retry Step are all role-gated. Read-only
  super_admin (Sam) sees no destructive or write-action buttons. Defense-
  in-depth: backend `requireSuperAdmin` middleware also enforces (plan §7.0).
- **URL state preservation** — all list views (Tenants, Plugins, Logs)
  persist filters + pagination + sort in URL search params. Navigation
  away and back preserves context (data-presentation §2.3).
- **`aria-live` announcements** — every async status update (refresh
  timestamps, saga progress, result counts, copy confirmations, deletion
  completion) is announced via `aria-live="polite"` (or `assertive` for
  critical health-down).
- **Error recovery on every dead end** — every empty/error state has a
  clear recovery action: "Clear filters", "Retry", "Reload", "Back to
  list", "Widen time range". No dead ends without recovery (user-journey.md
  dead-end table).
- **No `window.confirm()` or `window.alert()`** — all confirmations use
  `@plexica/ui Dialog` components (Constitution §frontend patterns, Rule 3).
- **No color alone** — every status badge, level badge, health status has
  icon (Lucide) + text + color (WCAG 1.4.1).

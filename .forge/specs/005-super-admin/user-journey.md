## Personas

### Persona 1: Dana — Super Admin (Platform Operator)

| Field | Value |
|-------|-------|
| Role | Platform Operator / Super Admin |
| Goal | Keep the platform healthy, onboard tenants fast, act decisively on incidents |
| Pain points | (1) Manual infra steps (SQL, Keycloak CLI, MinIO client) are slow and error-prone; (2) no visibility into cross-tenant state during incidents; (3) GDPR deletion anxiety — "did I really erase everything?" |
| Tech literacy | High |
| Device | Desktop (1440px), dual-monitor common |
| Accessibility needs | None (designs still meet WCAG 2.1 AA for all users) |
| Quote | "I need to provision a tenant in under a minute and know — not hope — that deletion left no traces." |

### Persona 2: Sam — Tenant Admin (Occasional Viewer)

| Field | Value |
|-------|-------|
| Role | Tenant Admin with view-only platform access (read-only super_admin role) |
| Goal | Understand why their tenant was suspended, check if an incident affects their users |
| Pain points | (1) No visibility into platform-level decisions that affect their tenant; (2) technical jargon in status pages is hard to parse; (3) doesn't know who to contact during outages |
| Tech literacy | Medium |
| Device | Desktop (laptop, 1280px) |
| Accessibility needs | Low vision (relies on zoom + high contrast) |
| Quote | "I just need to know if the platform is down or if it's just us." |

> **Design implication**: Dana is the primary persona — every screen optimizes for fast, confident, destructive-capable workflows. Sam is secondary — the read-only views (Dashboard, Health, Kafka) must use plain language and avoid infrastructure jargon. Sam never sees provisioning, suspend, or delete actions (role-gated UI, per FR-005 requireSuperAdmin).

## Journey 0: Dana signs in, recovers an expired session, and signs out

**Trigger**: Dana opens an admin route without a valid tab-scoped session.

```
Steps:
  1. Open /dashboard                    → Guard routes once through /login
  2. /login starts PKCE S256            → Browser redirects to master-realm Keycloak
  3. Authenticate under realm policy    → Keycloak handles password/MFA/SSO
  4. Return to /callback                → App validates state and exchanges code
  5. Land on /dashboard                 → Authenticated shell renders
  6. [Later] Refresh is no longer valid → "Session expired" is announced
  7. After the notice                   → Fresh PKCE starts; no inline form
  8. Choose Logout                      → Local state clears; visit Keycloak logout
  9. Keycloak ends current SSO session  → Return to exact registered /login

Outcome: Credentials stay on Keycloak; callback, expiry, and logout recover
without a password grant in Plexica.
```

**Edge Cases**:
- **Repeated sign-in**: duplicate effects/clicks do not replace the in-flight
  state/verifier pair.
- **Invalid/replayed state**: no exchange; show localized error and retry.
- **Keycloak unavailable**: show localized retry; expose no raw OAuth details.
- **Refresh failure**: clear all tokens, preserve only expiry UX state, then
  re-authenticate.
- **Logout**: use the ID token and exact environment post-logout URI; do not
  merely clear local state.

## Journey 1: Dana provisions a new tenant (005-04)

**Trigger**: Sales closes a deal; Dana receives a request to onboard "Acme Corp".

```
Steps:
  1. Click "Provision Tenant" in sidebar            → Wizard opens at Step 1
  2. Enter slug "acme", name "Acme Corp", admin email → Real-time slug validation (kebab-case, uniqueness)
  3. Click "Review"                                  → Step 2: summary card shows schema name, realm name, bucket name
  4. Click "Provision"                               → Step 3: progress stepper (schema → realm → bucket → seed)
  5. Wait ~20s                                       → Stepper completes, temp password displayed
  6. Copy temp password, click "Done"                → Redirect to tenant detail page

Outcome: Tenant operational; Dana sends credentials to Acme admin.
```

**Emotional Map**: 😐 Neutral (routine) → 🤔 Curious (slug available?) → 😊 Relieved (wizard is guided) → ⏳ Anxious (waiting for infra) → 🎉 Satisfied (temp password ready) → 🎯 Decisive (send credentials)

**Edge Cases**:
- **Error — slug already exists**: Step 1 input shows inline error "Slug 'acme' is already in use by another tenant". Dana must pick a different slug. (409 CONFLICT, conflictType: schema_exists / realm_exists / bucket_exists)
- **Error — realm already exists (orphaned from failed attempt)**: Step 3 stepper halts at "realm" step with red error. Dialog offers "Adopt existing resources" or "Force cleanup and retry". (per spec §7 edge cases)
- **Empty — no tenants yet**: Not applicable (provisioning creates the first).
- **Permission denied**: Sam (read-only) does not see the "Provision" nav item at all.

## Journey 2: Dana suspends and reactivates a tenant (005-05 / 005-06)

**Trigger**: Acme Corp is 60 days overdue on payment. Finance asks Dana to suspend access.

```
Steps:
  1. Navigate to Tenants list, search "acme"        → Filtered to 1 result
  2. Click tenant row                                → Tenant detail page, Info tab
  3. Click "Suspend" action button                   → Confirm dialog opens: "Suspend Acme Corp? All users will be immediately blocked."
  4. Click "Confirm Suspend"                         → Mutation submits with optimistic lock version; dialog shows loading spinner
  5. Dialog closes, status badge updates to "Suspended" (amber) → Toast: "Tenant suspended. Propagation < 5s."
  6. [Later] Payment received. Return to tenant detail.
  7. Click "Reactivate"                              → Confirm dialog: "Reactivate Acme Corp? Users will regain access."
  8. Click "Confirm Reactivate"                      → Status badge updates to "Active" (green)

Outcome: Tenant access cleanly cut and restored; audit log records both actions.
```

**Emotional Map**: 😐 Neutral → 🔍 Investigating (find the tenant) → 🤔 Decisive (confirm suspension) → ⏳ Anxious (is propagation really atomic?) → 😊 Relieved (badge confirms) → 😊 Relieved (reactivation is symmetric)

**Edge Cases**:
- **Error — concurrent action (409 CONFLICT)**: Another admin suspended Acme between Dana's page load and click. Dialog shows "This tenant was modified by another admin. Please reload and try again." Dana clicks "Reload" → re-reads state → sees already suspended.
- **Error — tenant already suspended**: "Suspend" button is disabled (grayed out) with tooltip "Tenant is already suspended". Reactivate button is enabled instead.
- **Dead end — filtered to empty**: Dana searches "acme" but typo'd "acm". Empty state: "No tenants match 'acm'. [Clear search]". Recovery: clear search or correct typo.

## Journey 3: Dana deletes a tenant with GDPR confirmation (005-07)

**Trigger**: Acme Corp churns and requests full data deletion per GDPR Article 17.

```
Steps:
  1. Navigate to tenant detail for "acme"
  2. Click "Delete Tenant" (destructive button, red) → TypeToConfirmDialog opens
  3. Read warning: "This permanently deletes schema, realm, and bucket. This cannot be undone."
  4. Type the tenant slug "acme" exactly             → Confirm button enables only when input === slug
  5. Click "Delete Permanently"                      → 202 Accepted; redirect to Deletion Status panel
  6. Watch deletion saga stepper: schema_drop → realm_delete → bucket_delete
  7. All 3 steps show "done" (green check)           → Tenant status: "Deleted" (gray); toast: "GDPR deletion complete"

Outcome: All three resources verified gone; audit log entry with completion timestamp.
```

**Emotional Map**: 😤 Tense (destructive action) → 😵 Overwhelmed (type-to-confirm forces deliberation) → 🎯 Decisive (typed correctly, clicking) → 🔍 Investigating (watching saga progress) → 💡 Insight (all green) → 😊 Relieved (GDPR obligation met)

**Edge Cases**:
- **Error — schema drop fails (active connections)**: Stepper shows "schema_drop" step as "failed" (red) with `lastError` message. "Retry Step" button appears. Dana clicks retry → backoff retries → eventually succeeds or stays failed for manual intervention. (per spec §7)
- **Error — realm delete fails (Keycloak unreachable)**: Same saga pattern — step marked "failed", retry available. Schema step stays "done"; bucket step stays "pending". Tenant remains "pending_deletion" until ALL steps complete.
- **Partial failure — saga stuck**: Deletion Status panel shows stuck step with attempts count and last error. Dana can retry individual steps without restarting the saga. The tenant is NEVER marked "deleted" until all 3 steps are "done".
- **Dead end — type-to-confirm mismatch**: Dana types "Acme" (wrong case). Confirm button stays disabled. Helper text: "Type 'acme' exactly to confirm." No ambiguity.
- **Permission denied**: Sam (read-only) sees no "Delete" button.

## Journey 4: Dana reviews and publishes a plugin (005-08)

**Trigger**: A plugin developer submits "CRM Pro" for review. It's in the review queue.

```
Steps:
  1. Navigate to Plugins page                        → Plugin catalog table with "Review Queue" filter tab
  2. Click "Pending Review" filter chip              → Table filters to reviewStatus = "pending"
  3. Click "CRM Pro" row                             → Plugin review dialog: manifest, description, installed count (0), versions
  4. Review manifest fields                          → All valid (Zod schema passed at submission)
  5. Click "Approve" with optional notes             → reviewStatus → "approved"; toast: "Plugin approved. Ready to publish."
  6. Click "Publish" on the plugin row               → Status → "published"; toast: "Plugin now visible in tenant marketplaces."
  7. [Later] Need to unpublish. Click "Unpublish".
  8. Warning dialog: "Installed by 3 tenants. Unpublishing will set status to 'deprecated' — existing installs keep working, but no new installs." → Click "Confirm"

Outcome: Plugin vetted, published, and later safely deprecated.
```

**Emotional Map**: 🤔 Curious (new plugin) → 🔍 Investigating (manifest review) → 🎯 Decisive (approve) → 😊 Satisfied (published) → 🤔 Cautious (unpublish with installs) → 😊 Relieved (deprecated, not broken)

**Edge Cases**:
- **Error — broken manifest at publish**: If manifest validation fails at publish (shouldn't happen post-approval, but defense-in-depth), 400 VALIDATION_ERROR with per-field errors. Plugin stays "draft". Dana sees field-level errors in the dialog.
- **Empty — no pending reviews**: Review queue tab shows "No plugins pending review" empty state with illustration.
- **Dead end — unpublish plugin with no installs**: Status → "unpublished" (not "deprecated"). Distinct semantic: "never installed" vs "was installed, now withdrawn". Dana sees the correct status badge.
- **Permission denied**: Sam sees catalog read-only; no approve/publish/unpublish buttons.

## Journey 5: Dana monitors system health (005-09 / 005-11)

**Trigger**: Dana starts the workday; checks platform status first thing.

```
Steps:
  1. Navigate to Health page                         → Service health grid: PostgreSQL, Keycloak, Redis, Kafka, MinIO
  2. Scan 5 service cards                            → All green (healthy) except Kafka: "degraded" (amber)
  3. Click Kafka card                                → Navigates to Kafka Status page
  4. View consumer lag table                         → Plugin "crm-pro" on tenant "acme" shows lag 1,450 (red, threshold 1000)
  5. View DLQ summary                                → "crm-pro" DLQ: 12 messages (amber, threshold 100)
  6. Decision: lag is growing. Dana notes it for investigation → Logs page to find related errors

Outcome: Degraded Kafka detected; Dana pivots to log investigation (Journey 6).
```

**Emotional Map**: 😐 Neutral (routine check) → 🤔 Alert (amber Kafka) → 🔍 Investigating (drill into Kafka) → 😤 Concerned (lag > threshold) → 🎯 Decisive (pivot to logs)

**Edge Cases**:
- **Error — health endpoint down**: If `/admin/health` itself fails (API unreachable), Health page shows error state: "Cannot reach health endpoint. Check core API status." with retry button.
- **Partial failure — one service down**: Only the down service shows red; others remain green. Dana sees the problem immediately without losing context of healthy services.
- **Empty — Kafka has no consumers**: Kafka Status table shows "No active consumers" empty state. Not an error — means no plugins installed yet.
- **Data emotion — overwhelmed**: If 3+ services are degraded/down, Dana may feel overwhelmed. Design mitigation: Health page header shows aggregate status ("2 of 5 services degraded") so Dana gets the headline before the detail.

## Journey 6: Dana investigates an incident via logs (005-10)

**Trigger**: Kafka lag is growing (from Journey 5). Dana needs to find the root cause.

```
Steps:
  1. Navigate to Logs page                           → Log viewer with filter bar (tenant, level, time range)
  2. Set filters: tenant = "acme", level = "error", time range = "last 1 hour"
  3. Click "Search"                                  → Log entries stream in (Loki proxy), newest first
  4. Scan log entries                                → Entry at 09:42: "Plugin proxy timeout for crm-pro on tenant acme"
  5. Click log entry                                 → Expandable row shows full metadata (requestId, duration, stack trace)
  6. Pivot: clear tenant filter, set level = "error" → See if other tenants affected
  7. Only acme errors appear                         → Conclusion: isolated to acme's crm-pro install, not platform-wide

Outcome: Root cause isolated; Dana contacts acme admin or disables the plugin.
```

**Emotional Map**: 🔍 Investigating (filters set) → 🤔 Focused (scanning entries) → 💡 Insight (timeout error found) → 🔍 Comparing (pivot to all tenants) → 💡 Insight (isolated to acme) → 🎯 Decisive (action: disable plugin or contact admin)

**Edge Cases**:
- **Error — Loki unreachable (503 SERVICE_UNAVAILABLE)**: Logs page shows "Log service is unavailable. The Loki data source may be down." with retry button. Dana can still navigate to other pages.
- **Error — Loki query timeout (503 LOG_QUERY_TIMEOUT)**: "Log query timed out. Try narrowing the time range or reducing the limit." Suggests recovery action.
- **Empty — no logs match filters**: "No log entries match your filters. [Clear all filters]" — clear recovery, not a dead end.
- **Dead end — too many results**: If limit=100 returns 100 results, show "Showing 100 of more results. Narrow your time range or increase specificity." Prevents Dana from thinking 100 is the total.
- **Filter persistence**: When Dana navigates away and returns, filters are preserved in URL search params. She doesn't lose her investigation context.
- **PII safety**: All log entries are redacted by Pino redact paths (email, password, token, secret). Dana never sees PII in logs (Constitution §Security).

## Cross-Journey Patterns

### Drill-down loops
Dana frequently moves: Dashboard → (metric anomaly) → Tenant List → Tenant Detail → (action) → back to Dashboard. The sidebar persists across all views, and "back" preserves filters/scroll (per data-presentation §2.3).

### Decision points → actions
Every data view ends with an actionable CTA:
- Dashboard → "View tenants" / "Check health"
- Health page → "View Kafka details" (if Kafka degraded)
- Kafka page → "View logs" (if lag high)
- Logs page → "View tenant detail" (clicking tenant in log entry)
- Tenant detail → Suspend / Reactivate / Delete / View audit

### Dead ends + recoveries
| Dead end | Recovery |
|----------|----------|
| Filtered tenant list to empty | "Clear search" + "Clear filters" buttons |
| Logs query no match | "Clear all filters" button |
| Deletion saga stuck | "Retry step" button per failed step |
| Health endpoint unreachable | "Retry" button + message to check core API |
| Provisioning slug conflict | Inline error + suggestion to pick different slug |
| Concurrent action 409 | "Reload" button to re-read latest state |

### Data emotions summary
🤔 Curious (exploring dashboard) · 🔍 Investigating (logs, Kafka lag) · 😵 Overwhelmed (type-to-confirm for deletion) · 💡 Insight (root cause found) · 🎯 Decisive (suspend/delete/publish) · 😤 Frustrated (saga stuck, Loki down) · 😊 Relieved (saga complete, propagation confirmed)

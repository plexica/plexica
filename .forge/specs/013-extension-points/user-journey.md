## 2. User Journeys

### Journey 1: Mei — Declares a Slot and Contributes to Another Plugin's Slot

**Trigger**: Mei is building a Billing plugin that needs to add a "Create Invoice"
action button inside the CRM plugin's contact detail view. She also wants her own
Billing detail view to be extensible by other plugins.

**FR Coverage**: FR-001, FR-002, FR-003, FR-005, FR-009, FR-027, FR-028, FR-030

**Steps:**

1. **Discover available slots** → Mei opens the Extension Registry docs or calls
   `GET /api/v1/extension-registry/slots` to find CRM's `contact-detail-actions`
   slot with type `action` and its `contextSchema`.
   - 😐 Neutral — "Standard API call, straightforward."

2. **Review slot contract** → She reads the `contextSchema` (`{ contactId: string,
tenantId: string }`) and the `requiredPermission`
   (`plugin.crm.extend.contact-detail`).
   - 😊 Relieved — "Clear contract. I know exactly what props I'll receive."

3. **Declare own slots in manifest** → Mei adds `extensionSlots` to her Billing
   plugin's `PluginManifest`: `{ id: "invoice-detail-actions", type: "action",
contextSchema: { invoiceId: string } }`.
   - 😐 Neutral — "Simple manifest addition."

4. **Declare contribution targeting CRM slot** → She adds a `contributions` entry:
   `{ targetPluginId: "crm", targetSlotId: "contact-detail-actions",
type: "action", widgetName: "CreateInvoiceButton", priority: 50 }`.
   - 😐 Neutral — "Manifest-driven, no runtime code needed for registration."

5. **Build the contributed component** → Mei writes `CreateInvoiceButton.tsx`,
   receiving `PluginProps` + `{ contactId }` as context props. She uses the SDK's
   `useExtensionSlot` hook to test rendering.
   - 😤 Mild frustration — "I need to run both CRM and Billing locally to test.
     Wish there were a standalone slot inspector."

6. **Deploy and activate** → Plugin is activated. The Extension Registry syncs
   manifest, validates slot target exists, type matches, and registers the
   contribution.
   - 🎉 Satisfied — "Contribution appeared immediately in the CRM contact detail.
     Priority ordering is correct."

**Outcome**: Mei's Billing plugin contributes a "Create Invoice" button to CRM's
contact detail view, and her own Billing detail view is extensible by future plugins.

**Edge Cases from this journey:**

- **Error — Target slot doesn't exist yet**: CRM isn't installed. Contribution is
  stored as dormant (`validation_status: pending`). When CRM is installed later,
  contribution becomes visible. (Edge Case #1)
- **Error — Type mismatch**: Mei's contribution type is `panel` but the slot
  expects `action`. Flagged at activation as `type_mismatch`, excluded from
  rendering. Warning in admin settings. (Edge Case #7)
- **Empty state — No contributions to own slot**: Mei's Billing detail view has
  a `invoice-detail-actions` slot but no other plugins contribute yet. The
  `<ExtensionSlot>` renders nothing (no empty placeholder visible to users;
  the slot area collapses). Developer inspector shows "0 contributions".

### Journey 2: Raj — Manages Extension Visibility for His Workspace

**Trigger**: Raj's Sales workspace has 4 plugins active (CRM, Billing, HR, Analytics).
The HR plugin contributes an "Employee Details" panel to CRM's contact detail view,
but it's not relevant for the Sales team. Raj wants to hide it.

**FR Coverage**: FR-010, FR-011, FR-022, FR-025, US-003

**Steps:**

1. **Navigate to workspace settings** → Raj opens Workspace Settings → Extensions
   tab. He sees contributions grouped by host plugin, then by slot.
   - 😐 Neutral — "I can find the Extensions tab easily."

2. **Scan available contributions** → Under "CRM → Contact Detail Actions", Raj
   sees three contributions: "Create Invoice" (Billing, priority 50), "View
   Engagement" (Analytics, priority 100), and "Employee Details" (HR, priority 200).
   Each has a toggle, icon, and description.
   - 😊 Relieved — "Clear list. I can see which plugin each one comes from."

3. **Disable HR contribution** → Raj toggles off "Employee Details" for his
   workspace. Confirmation is immediate — no save button needed (toggle is
   auto-saved via PATCH).
   - 😊 Satisfied — "That was easy. Instant toggle."

4. **Verify in CRM view** → Raj navigates to a CRM contact detail. The HR panel
   is gone. The Billing and Analytics contributions remain.
   - 🎉 Delighted — "Exactly what I wanted. No page refresh needed."

5. **Notice tenant-disabled contribution** → The Analytics plugin also contributes
   a "Pipeline Forecast" panel, but it's grayed out with a tooltip: "Disabled by
   tenant admin. Contact your tenant administrator to enable." Raj cannot toggle it.
   - 😤 Mild frustration — "I understand why, but I wish I could request it."

**Outcome**: Raj's Sales workspace shows only the extensions relevant to his team.
Other workspaces are unaffected.

**Edge Cases from this journey:**

- **Error — Raj is not workspace admin**: The Extensions tab is visible but all
  toggles are disabled with tooltip "Admin access required". (FR-022 permission
  check → `WORKSPACE_VISIBILITY_DENIED`)
- **Error — Tenant admin disables entire plugin**: All contributions from that
  plugin cascade-disable. Raj's toggles are superseded. If the tenant admin
  re-enables the plugin, Raj's previous "off" setting is preserved (not
  auto-enabled). (Edge Case #6, FR-025)
- **Empty state — No extensions available**: The Extensions tab shows an empty
  state: "No extensions available. Extensions appear when plugins contribute to
  each other's views." (No CTA needed — admin cannot create extensions.)

### Journey 3: Ana — Uses Extended Plugin Views and Manages Extension Permissions

**Sub-Journey 3a: End User — Viewing CRM Contact with Extensions**

**Trigger**: Ana opens a CRM contact detail to prepare for a sales call.
Extensions from Billing and Analytics are visible in the contact detail view.

**FR Coverage**: FR-007, FR-008, FR-009, US-007, US-008, NFR-008, NFR-011

**Steps:**

1. **Navigate to CRM contact** → Ana clicks a contact in the CRM list. The
   contact detail page loads. The `<ExtensionSlot>` areas show skeleton
   placeholders while Module Federation loads the remote components.
   - 😐 Neutral — "Skeletons are loading. Expected."

2. **Extensions load successfully** → After ~300ms, the Billing "Create Invoice"
   button and Analytics "View Engagement" panel appear in their slot areas.
   Each has a subtle puzzle-piece icon (visible on hover) indicating it's from
   another plugin.
   - 😊 Pleased — "Quick load. I can see what I need."

3. **Click "Create Invoice"** → Ana clicks the action button. The Billing plugin
   opens an invoice creation flow, pre-populated with the contact's ID and name.
   - 🎉 Delighted — "No switching to the Billing plugin! This saved me 4 clicks."

4. **Analytics panel crashes** → The Analytics contribution throws a rendering
   error. A fallback card appears: "Extension unavailable — View Engagement from
   Analytics plugin could not load. [Dismiss]". The rest of the contact detail
   and the Billing button remain fully functional.
   - 😤 Mild frustration → 😌 Relieved — "At least the page didn't break. I can
     still work."

5. **Screen reader announces error** → Ana's screen reader (when used) announces
   "Alert: Extension unavailable. View Engagement from Analytics plugin could not
   load." via `role="alert"`.
   - 😊 Neutral — "Accessible error handling."

**Outcome**: Ana uses the enriched contact detail view. One extension failure does
not affect the rest of her workflow.

**Sub-Journey 3b: Tenant Admin — Managing Extension Permissions**

**Trigger**: Ana (also tenant admin) receives a request from the HR team to enable
the HR plugin's data extension on CRM contacts. She needs to grant the permission.

**FR Coverage**: FR-006, US-006, FR-026

**Steps:**

1. **Navigate to tenant admin** → Ana opens the Tenant Admin → Plugins → Extension
   Permissions page.
   - 😐 Neutral — "Finding the right settings page."

2. **View permission matrix** → Ana sees a list of plugins with their required
   extension permissions. The HR plugin shows: "Required permission:
   `plugin.crm.extend.contact-detail`" — Status: Not Granted.
   - 😐 Neutral — "Clear what's needed."

3. **Grant permission** → Ana clicks "Grant" next to the HR plugin's CRM extension
   permission. A confirmation dialog explains: "This allows HR plugin to contribute
   UI to CRM contact detail views across all workspaces. Workspace admins can
   further control visibility per workspace."
   - 😊 Relieved — "Good that I understand the scope before granting."

4. **Permission granted** → The HR plugin's contribution becomes active. It appears
   in all workspaces (unless workspace admins have disabled it).
   - 🎉 Satisfied — "HR team should now see their extension."

5. **Revoke permission later** → Ana revokes the permission for the HR plugin.
   Its contributions disappear from all slots without page reload (SSE/next
   navigation).
   - 😐 Neutral — "Clean and immediate."

**Outcome**: Ana controls which plugins can contribute to other plugins' views at
the tenant level, providing a security and trust boundary.

**Edge Cases from this journey:**

- **Error — New plugin installed with contributions**: The new plugin's
  contributions are inactive by default until Ana grants the required permission.
  She sees a notification in the admin panel: "New plugin 'Expense Tracker'
  requires extension permissions." (US-006 AC-3)
- **Error — Contributing plugin is disabled**: Its contributions cannot be
  re-enabled at tenant or workspace level. The permission row shows
  "Plugin disabled — enable plugin first."
- **Permission denied — Ana is not tenant admin**: The Extension Permissions
  page returns 403. Ana sees "Tenant admin access required."

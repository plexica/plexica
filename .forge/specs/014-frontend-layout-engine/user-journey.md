## 2. User Journeys

### Journey 1: Carla Configures Layout for the CRM Contact Form

**Persona**: Carla (Tenant Admin)
**Goal**: Reorder fields so "Phone" is first, hide "Internal Notes" from VIEWER and MEMBER roles, and make "Budget" read-only for MEMBER role.
**FR coverage**: FR-003, FR-012, FR-013, FR-014, US-001, US-002, US-003, US-008

**Trigger**: Carla receives complaints from sales reps that the contact form is cluttered and phone number is buried below email.

**Steps**:

| Step | Action                                                                                                                    | System Response                                                                                                                                                                                                                                                       | Emotional State                      |
| ---- | ------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------ |
| 1    | Navigates to **Settings > Layout Configuration** from the tenant admin sidebar                                            | Page loads. Form selector dropdown shows list of configurable forms from plugin manifests (FR-023)                                                                                                                                                                    | Neutral — routine admin task         |
| 2    | Selects **"CRM > Contact Form"** from the form selector dropdown                                                          | Config table loads with fields: First Name, Phone, Email, Budget, Internal Notes. Default order from plugin manifest. Sections: Contact Information, Billing Details                                                                                                  | Neutral — scanning the layout        |
| 3    | Clicks the **[Up] arrow** on "Phone" row twice to move it to position 1                                                   | Phone moves from position 2 to position 1. First Name shifts to position 2. Order numbers update. **Unsaved changes** indicator appears in the toolbar. Screen reader announces "Phone moved to position 1"                                                           | Satisfied — simple interaction       |
| 4    | In the **Visibility by Role** columns, clicks the toggle for "Internal Notes" × VIEWER column to set it to **hidden** (✗) | Toggle changes from ✓ (visible) to ✗ (hidden). Cell shows ✗ icon. Unsaved changes indicator persists                                                                                                                                                                  | Focused — configuring roles          |
| 5    | Clicks the toggle for "Internal Notes" × MEMBER column to set it to **hidden** (✗)                                        | Toggle changes from ✓ to ✗. Internal Notes is now hidden for both VIEWER and MEMBER                                                                                                                                                                                   | Confident — clear visual matrix      |
| 6    | Clicks the toggle for "Budget" × MEMBER column, cycles to **read-only** (✓ᵣ)                                              | Toggle cycles: visible → read-only → hidden → visible. Stops at read-only. Cell shows ✓ᵣ icon                                                                                                                                                                         | Neutral — learning the cycle         |
| 7    | Selects **"Preview as: MEMBER"** from the Preview dropdown                                                                | A read-only preview panel appears below the config table showing the form as MEMBER would see it: Phone, First Name, Email, Budget (greyed/read-only). Internal Notes is absent. Preview area has `aria-live="polite"` announcement: "Previewing form as MEMBER role" | Delighted — can verify before saving |
| 8    | Selects **"Preview as: VIEWER"** from the Preview dropdown                                                                | Preview updates: Phone, First Name, Email. Budget visible (no role-specific override for VIEWER — falls back to globalVisibility "visible"). Internal Notes absent                                                                                                    | Satisfied — verification complete    |
| 9    | Clicks **[Save]**                                                                                                         | Layout config saved via PUT endpoint. Success toast: "Layout configuration saved". Unsaved changes indicator clears                                                                                                                                                   | Relieved — changes are live          |

**Outcome**: The contact form is now configured. Phone is first, Internal Notes is hidden for VIEWER and MEMBER, and Budget is read-only for MEMBER.

**Edge Cases from this journey**:

- **Error: Save fails (network)** → Toast error: "Failed to save layout configuration. Please try again." Unsaved changes persist. Retry available.
- **Concurrent edit (409 CONFLICT)** → Dialog: "This configuration was modified by another admin. Reload to see their changes, or overwrite with yours?" with [Reload] and [Overwrite] buttons.
- **Empty state (no forms)** → EmptyState component: "No configurable forms available. Install a plugin with form schemas to get started." with link to Plugin Registry.

### Journey 2: Dev Opens the Configured Contact Form

**Persona**: Dev (Sales Representative, MEMBER role)
**Goal**: Open the CRM contact form and quickly enter a new contact during a phone call.
**FR coverage**: FR-025, FR-007, FR-008, FR-017, US-001, US-002, US-009

**Trigger**: Dev receives a phone call from a potential client and needs to create a new contact record quickly.

**Steps**:

| Step | Action                                                                               | System Response                                                                                                                                                                         | Emotional State                             |
| ---- | ------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------- |
| 1    | Navigates to **CRM > Contacts > New Contact**                                        | `<LayoutAwareForm>` component mounts. Calls `GET /api/v1/layout-configs/crm-contact-form/resolved?workspaceId=W1`. Response cached with `staleTime: 60s` (FR-017)                       | Neutral — starting data entry               |
| 2    | Sees the form with fields in order: **Phone, First Name, Email, Budget (read-only)** | Form renders with Phone field at top (per Carla's configuration). Budget field has a grey background and `disabled` attribute. "Internal Notes" field is completely absent from the DOM | Pleased — phone field is right where needed |
| 3    | Types phone number into the **Phone** field (already focused)                        | Standard input behavior. Field validates phone format on blur                                                                                                                           | Focused — on the call                       |
| 4    | Tabs to **First Name**, types the name                                               | Standard text input. Tab order follows configured field order                                                                                                                           | Efficient — fast data entry                 |
| 5    | Tabs to **Email**, types email address                                               | Standard email input. Validates email format on blur                                                                                                                                    | Neutral                                     |
| 6    | Notices **Budget** is greyed out (read-only)                                         | Budget shows "0" (default value). Input is visually disabled with read-only styling. Tooltip on focus: "This field is read-only for your role"                                          | Understanding — no confusion                |
| 7    | Clicks **[Save Contact]**                                                            | Form submits. Budget value (default 0) and Internal Notes (hidden, uses default if any) are handled server-side. Success feedback: "Contact saved"                                      | Satisfied — quick and clean                 |

**Outcome**: Dev enters the contact in under 30 seconds. Only relevant fields are shown. No accidental edits to sensitive fields.

**Edge Cases from this journey**:

- **Layout config unavailable (fail-open, NFR-008)** → Form renders with plugin manifest defaults. All fields visible in manifest order. No error shown to user. Warning logged to console.
- **All fields hidden (Edge Case #3)** → Empty state: "No fields are visible for your role. Contact your administrator." Form cannot be submitted.
- **Plugin uninstalled mid-session (Edge Case #8)** → Fields from uninstalled plugin gracefully removed. Remaining fields still work.

### Journey 3: Monica Sees Additional Fields as Manager

**Persona**: Monica (Regional Manager, ADMIN team role)
**Goal**: Review a contact record with full financial details (budget, commission) and internal notes that her sales reps cannot see.
**FR coverage**: FR-025, FR-007, FR-008, US-002, US-003

**Trigger**: Monica needs to review a high-value contact before a meeting to understand the budget and read internal notes left by the previous manager.

**Steps**:

| Step | Action                                                          | System Response                                                                                                                                                         | Emotional State                 |
| ---- | --------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------- |
| 1    | Navigates to **CRM > Contacts** and opens a contact record      | `<LayoutAwareForm>` resolves layout for ADMIN role. Fields: Phone, First Name, Email, Budget (**editable** — ADMIN has ✓ₑ), Internal Notes (**visible** — ADMIN has ✓ₑ) | Neutral — routine review        |
| 2    | Sees **Budget** field as an editable input with value "$45,000" | Budget field is fully interactive (not greyed out like Dev sees). Standard input with currency formatting                                                               | Satisfied — has the data needed |
| 3    | Scrolls to **Internal Notes** section                           | Internal Notes textarea visible with content: "High-priority lead. Referred by existing client. Discount authorized up to 10%."                                         | Informed — critical context     |
| 4    | Edits the Internal Notes to add: "Meeting scheduled March 15"   | Standard textarea edit. Form enters unsaved state                                                                                                                       | Productive                      |
| 5    | Clicks **[Save]**                                               | Form submits with updated Internal Notes. Budget value preserved (editable for ADMIN). Success toast                                                                    | Confident — data is complete    |
| 6    | Navigates to **CRM > Contacts** list view                       | `<LayoutAwareTable>` renders with columns configured for ADMIN: Name, Email, Phone, Budget, Owner. (Dev's MEMBER view would hide Budget per column config)              | Efficient — full overview       |

**Outcome**: Monica sees the full picture — budget, internal notes, and all columns — while her sales reps see a streamlined view. Same form, different role, different experience.

**Edge Cases from this journey**:

- **Monica has dual team membership** → She is ADMIN in Team A and MEMBER in Team B. Per FR-007, most permissive wins: she sees Budget as editable (ADMIN permission takes precedence).
- **Monica has only Keycloak realm role (no team membership, Edge Case #2)** → Visibility resolves using Keycloak realm role only. `globalVisibility` applies. Team role-specific rules ignored.
- **Read-only field bypass via API (US-003 AC3)** → Even if someone crafts a request to modify a read-only field, the backend strips it before reaching the service layer (FR-021).

### Journey 4: Carla Accidentally Hides a Required Field

**Persona**: Carla (Tenant Admin)
**Goal**: Hide the "Email" field (which is required with no default value) for VIEWER role — but the system warns her about the risk.
**FR coverage**: FR-010, FR-011, US-007

**Trigger**: Carla wants to simplify the VIEWER experience further by hiding the Email field, not realizing it's marked as required in the plugin manifest.

**Steps**:

| Step | Action                                                                                                                | System Response                                                                                                                                                                                                                                                       | Emotional State                     |
| ---- | --------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------- |
| 1    | In the Layout Config admin panel, Carla clicks the visibility toggle for "Email" × VIEWER to set it to **hidden** (✗) | Toggle changes to ✗. An orange warning icon (⚠) appears next to "Email" row because it's a required field. Unsaved changes indicator appears                                                                                                                          | Neutral — making a change           |
| 2    | Clicks **[Save]**                                                                                                     | PUT request sent. Backend returns **400 REQUIRED_FIELD_NO_DEFAULT** error. The admin UI intercepts this specific error code and opens a **confirmation dialog**                                                                                                       | Surprised — didn't expect a warning |
| 3    | Reads the confirmation dialog                                                                                         | Dialog content: **"Required Field Warning"** — "The field 'Email' is required by the CRM plugin and has no default value. Hiding it for the VIEWER role may cause form submission errors for users with that role. Do you want to proceed?" [Cancel] [Proceed Anyway] | Cautious — understanding the risk   |
| 4    | **Option A: Clicks [Cancel]**                                                                                         | Dialog closes. Toggle reverts to previous state (✓ visible). Unsaved changes for this specific toggle are discarded. Toast: "Change cancelled"                                                                                                                        | Relieved — avoided a mistake        |
| 4'   | **Option B: Clicks [Proceed Anyway]**                                                                                 | Client resends PUT with `acknowledgeWarnings: true`. Backend accepts. Config saved. Success toast: "Layout configuration saved (with warnings)". Override recorded in audit log (ADR-025) with `action: 'layout_config.required_field_warning_overridden'`            | Resolved — made an informed choice  |

**Outcome (Option A)**: Carla avoids hiding a required field. No disruption to end users.
**Outcome (Option B)**: Carla explicitly acknowledges the risk. The system saves the config and creates an audit trail.

**Edge Cases from this journey**:

- **Required field WITH default value (FR-010)** → No warning dialog. Field is hidden, form auto-populates with default value on submission. Smooth experience.
- **Multiple required fields hidden at once** → Dialog lists ALL affected fields: "The following required fields have no default value: Email, Status. Hiding them may cause submission errors."
- **Revert after saving (US-010)** → Carla can click [Revert to Previous Version] to undo the change. The previous config is restored. Audit log records the revert.

### Scope

This design spec covers 4 UI components:

1. **Layout Config Admin Panel** — Tenant/workspace admin configuration interface (FR-012, FR-013, FR-014)
2. **`<LayoutAwareForm>`** — End-user form wrapper with layout transformations (FR-025)
3. **`<LayoutAwareTable>`** — End-user table wrapper with column visibility (FR-026)
4. **Required Field Confirmation Dialog** — Warning dialog for hiding required fields (FR-011)

**Notes**:

- Sidebar collapses to hamburger at `< 1024px` (per existing shell behavior)
- On tablet, content scrolls vertically. All tables are full-width
- The visibility toggle cells use icon buttons, not checkboxes — each cell cycles through: ✓ₑ (editable) → ✓ᵣ (read-only) → ✗ (hidden) → ✓ₑ (editable)
- The "Global" column is a `<Select>` dropdown: visible / hidden / readonly
- [▲] arrow is disabled on the first row; [▼] disabled on the last row
- The "⚠ required field" indicator appears as an orange warning icon next to the field name for fields with `required: true` in the manifest
- "(!) stale field" indicator appears as a yellow badge when a field in the config no longer exists in the plugin manifest (Edge Case #1)
- The form selector dropdown is populated from `GET /api/v1/layout-configs/forms` (FR-023)
- Scope selector switches between Tenant and Workspace views. When Workspace is selected, a workspace picker appears

#### State Inventory

| State               | Trigger                                    | Visual Change                                                                                                             | User Feedback                                       |
| ------------------- | ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------- |
| Default (no config) | First visit, no layout config exists       | Table shows fields in plugin manifest order. All visibility toggles default to ✓ₑ (editable). Global = visible            | "Using plugin defaults. Make changes to customize." |
| Configured          | Existing layout config loaded              | Table shows configured order, visibility overrides reflected in toggle states                                             | —                                                   |
| Unsaved changes     | Any toggle or order change                 | Orange banner: "⚠ Unsaved changes" above action buttons. [Save] button becomes primary. [Discard] appears                 | Browser `beforeunload` warning if navigating away   |
| Saving              | [Save] clicked                             | [Save] button shows spinner + "Saving…". All inputs disabled                                                              | —                                                   |
| Save success        | PUT returns 200                            | Success toast: "Layout configuration saved". Unsaved changes banner clears. [Revert to Previous] becomes enabled          | Toast auto-dismisses after 5s                       |
| Save error          | PUT returns 4xx/5xx                        | Error toast: "Failed to save layout configuration." Unsaved changes persist                                               | Toast persists until dismissed                      |
| Conflict (409)      | PUT returns 409 LAYOUT_CONFIG_CONFLICT     | Dialog: "This configuration was modified by another admin. Reload or overwrite?" [Reload] [Overwrite]                     | Dialog blocks further action until resolved         |
| Stale fields        | Plugin manifest updated after config saved | Yellow "(!) stale" badge next to affected fields. Banner: "Some fields are no longer available in the plugin"             | Non-blocking warning                                |
| No forms available  | No plugins with formSchemas installed      | EmptyState: illustration + "No configurable forms available. Install a plugin with form schemas." [Go to Plugin Registry] | —                                                   |

#### Responsive Notes

- **1440px**: Full sidebar + content area. Config table, section order, column config, and preview all visible without horizontal scroll.
- **1024px**: Sidebar collapsed. Content area full width. Tables may need horizontal scroll if many roles are configured.
- **768px**: Sidebar behind hamburger. Single-column layout. Tables scroll horizontally if needed. Preview panel stacks below config tables.
- **375px**: Not a primary target for admin panel (per Constitution Art. 1.3, admin interfaces are desktop-first). Functional but tables scroll horizontally. Buttons stack vertically.

### Screen 3: LayoutAwareForm (End-User)

**Source**: FR-025, FR-007, FR-008, FR-010, FR-017 | **Type**: New

The `<LayoutAwareForm>` is a wrapper component that wraps any existing plugin
form. It fetches the resolved layout config for the current user via
`GET /api/v1/layout-configs/:formId/resolved` and applies:

1. Field ordering (per configured order)
2. Field visibility (hidden fields removed from DOM)
3. Read-only enforcement (disabled inputs with visual treatment)
4. Section ordering (sections rendered in configured order)
5. Default value injection for hidden required fields (FR-010)

**Source**: FR-026, FR-005, FR-008 | **Type**: New

### Component: LayoutAwareForm

| Property    | Value                          |
| ----------- | ------------------------------ |
| Type        | Wrapper component (HOC-like)   |
| Used on     | Screen 3 (end-user form view)  |
| FR coverage | FR-025, FR-007, FR-008, FR-010 |
| Existing?   | No (new)                       |

| Prop          | Type                                                     | Required | Description                                       |
| ------------- | -------------------------------------------------------- | -------- | ------------------------------------------------- |

8. Hidden required fields with defaults: auto-injects `defaultValue` in form state

**States**:

| Resolved | Form with layout applied    | `aria-busy="false"` |

## 6. Interaction Patterns

### 6.1 Visibility Toggle Cycling

The `VisibilityToggle` component uses a **tri-state cycle** (field mode) or
**bi-state cycle** (column mode). This avoids the complexity of a dropdown per
cell while keeping the table compact.

**Field mode cycle** (click or Enter/Space):

```
✓ₑ (editable, green) → ✓ᵣ (read-only, blue) → ✗ (hidden, red) → ✓ₑ (editable, green)
```

**Column mode cycle** (click or Enter/Space):

```
✓ (visible, green) → ✗ (hidden, red) → ✓ (visible, green)
```

**Screen reader announcement on each cycle**:
`"{field name} visibility for {role} changed to {new state}"`

### 6.2 Field Reordering (Up/Down Arrows)

Per FR-013 and the spec's explicit scope statement, V1 uses up/down arrow
buttons only (no drag-and-drop).

**Click [▲]**: Field swaps with the field above it. Both order numbers update.
**Click [▼]**: Field swaps with the field below it.
**Boundary**: [▲] disabled on first row, [▼] disabled on last row.
**Screen reader**: `aria-live="polite"` region announces "Phone moved to position 1 of 5".
**Keyboard**: Tab to arrow button, Enter/Space to activate.

### 6.3 Preview Mode

The "Preview as Role" dropdown triggers a client-side computation (no API call)
that filters and reorders fields based on the current unsaved config state.
Changes in the config table reflect immediately in the preview.

**Behavior**:

1. Select a role from the dropdown
2. Preview panel computes visible fields for that role
3. Fields render in configured order with read-only treatment where applicable
4. `aria-live="polite"` announces "Previewing form as MEMBER role"

### 6.4 Unsaved Changes Warning

When the admin has unsaved changes and attempts to navigate away:

- `beforeunload` browser event fires with standard warning
- TanStack Router's `onBeforeNavigate` hook shows a confirmation dialog
- The unsaved changes banner persists until Save or Discard

### 6.5 Save Flow (with Required Field Check)

```
Admin clicks [Save]
  ↓
PUT /api/v1/layout-configs/:formId
  ↓
200 OK → Success toast → Clear unsaved state
  ↓ (or)
400 REQUIRED_FIELD_NO_DEFAULT → Open RequiredFieldWarningDialog
  ↓
  [Cancel] → Close dialog, revert toggles
  [Proceed Anyway] → Re-send PUT with acknowledgeWarnings: true
    ↓
    200 OK → Close dialog, success toast (with warning icon)
  ↓ (or)
409 CONFLICT → Show conflict dialog (Reload / Overwrite)
```

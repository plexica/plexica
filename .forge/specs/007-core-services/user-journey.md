### Journey 1: Dana — Global Search for a Contact

**Trigger**: Dana needs to find a CRM contact named "John" to check their latest deal status. She is on the workspace dashboard.

**Preconditions**:

- Dana is logged in to her tenant (Acme Corp)
- CRM plugin is installed and has indexed contacts via the Search Service
- Dana has `crm:contacts:read` permission

#### Happy Path

| Step | User Action                                                            | System Response                                                                                               | Emotional State                     |
| ---- | ---------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- | ----------------------------------- |
| 1    | Clicks the search bar in the header (or presses `/` keyboard shortcut) | Search bar receives focus; placeholder reads "Search across all data..."                                      | 😐 Neutral — knows where to look    |
| 2    | Types "john"                                                           | After 300ms debounce, shows real-time results grouped by type: **Contacts (3)**, **Deals (1)**, **Files (2)** | 😊 Relieved — results appear fast   |
| 3    | Scans results; sees "John Smith — Acme Sales" under Contacts           | Each result shows type icon, name, and snippet with match highlighted                                         | 😊 Confident — right person found   |
| 4    | Clicks "John Smith — Acme Sales"                                       | Navigates to CRM contact detail page; search overlay closes                                                   | 🎉 Satisfied — arrived in 4 seconds |

**Outcome**: Dana found the contact in under 5 seconds without leaving the page she was on. The search covered all plugin data indexed via the Search Service (per FR-011, FR-013, FR-014).

**Delight opportunities**:

- **Step 2**: Show recent searches as suggestions before typing (smart defaults)
- **Step 3**: Keyboard navigation with arrow keys to move between results; Enter to select
- **Step 4**: Preserve search query in URL for share/bookmark capability

| Step | User Action                          | System Response                                                                                                                      | Emotional State      |

### Journey 2: Dana — Uploading a File Attachment

**Trigger**: Dana needs to attach a signed contract PDF (2.5 MB) to a deal record in the CRM plugin.

**Preconditions**:

- Dana is on the CRM deal detail page
- CRM plugin uses the Storage Service for file attachments (per FR-001, FR-002)
- Dana has `crm:deals:write` permission

#### Happy Path

| Step | User Action                                  | System Response                                                                                                                                               | Emotional State                       |
| ---- | -------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------- |
| 1    | Clicks "Attach File" button on the deal page | File picker dialog opens (native OS dialog)                                                                                                                   | 😐 Neutral                            |
| 2    | Selects `contract-signed.pdf` (2.5 MB)       | Upload begins immediately. Progress bar appears below the button: `[████████░░░░░░░░░░░░] 42% — contract-signed.pdf (1.05 / 2.5 MB)` with a Cancel (×) button | 😊 Relieved — can see progress        |
| 3    | Waits ~2 seconds for upload to complete      | Progress bar reaches 100%. Transitions to: `✓ contract-signed.pdf (2.5 MB) — Uploaded`                                                                        | 🎉 Satisfied — clear success feedback |
| 4    | Sees the file appear in the Attachments list | File shown with name, size, upload time, and Download link (via signed URL per FR-001)                                                                        | 🎉 Confident — file is saved          |

**Outcome**: File is stored in the tenant bucket `tenant-{tenant_id}` via Storage Service. Dana sees clear progress and confirmation. Per FR-001, FR-002.

**Delight opportunities**:

- **Step 2**: Drag-and-drop support in addition to file picker
- **Step 3**: Subtle animation on success (checkmark appears)
- **Step 4**: File thumbnail preview for images

---

#### Edge Case A: File Too Large

**Trigger**: Dana tries to upload `presentation.zip` (150 MB) which exceeds the tenant's 100 MB limit (Edge Case #1 from spec).

| Step | User Action                              | System Response                                                                                               | Emotional State                      |
| ---- | ---------------------------------------- | ------------------------------------------------------------------------------------------------------------- | ------------------------------------ |
| 1    | Selects `presentation.zip` (150 MB)      | Client-side validation catches oversize immediately (before upload starts)                                    | 😐 Neutral                           |
| 2    | —                                        | Error message below Attach File button: `⚠ File too large. Maximum file size is 100 MB. Your file is 150 MB.` | 😤 Frustrated — but knows what to do |
| 3    | Compresses file or selects a smaller one | Upload proceeds normally                                                                                      | 😊 Recovered                         |

**Recovery path**: Clear error with exact limits. No partial upload attempted.
**Feedback**: "File too large. Maximum file size is 100 MB. Your file is 150 MB."

---

#### Edge Case B: Upload Failure (Storage Unavailable)

**Trigger**: MinIO is temporarily unavailable during upload (Edge Case #2 from spec).

| Step | User Action                    | System Response                                                                                                      | Emotional State       |
| ---- | ------------------------------ | -------------------------------------------------------------------------------------------------------------------- | --------------------- |
| 1    | Selects file and upload starts | Progress bar begins normally                                                                                         | 😐 Neutral            |
| 2    | —                              | Upload fails at 60%. Progress bar turns red: `✗ Upload failed — contract-signed.pdf. [Retry]`                        | 😡 Frustrated         |
| 3    | Clicks "Retry"                 | System retries upload from beginning. Progress bar resets                                                            | 😐 Cautiously hopeful |
| 4a   | Retry succeeds                 | Success state shown                                                                                                  | 😊 Relieved           |
| 4b   | Retry fails 3 times            | Error: `Upload failed after 3 attempts. The storage service may be temporarily unavailable. Please try again later.` | 😡 Needs admin help   |

**Recovery path**: Automatic retry up to 3 times with exponential backoff (per FR-003 / Edge Case #2). Manual retry button available.
**Feedback**: "Upload failed after 3 attempts. The storage service may be temporarily unavailable. Please try again later."

### Journey 3: Marco — Monitoring Background Jobs

**Trigger**: Marco wants to check why a nightly CSV export job hasn't completed. He suspects it failed overnight. He opens the Admin dashboard.

**Preconditions**:

- Marco is logged in as Tenant Admin
- Job Queue Service is active with jobs submitted by CRM plugin (per FR-007)
- Marco has `admin:jobs:read` permission

#### Happy Path

| Step | User Action                                              | System Response                                                                                                                  | Emotional State                           |
| ---- | -------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------- |
| 1    | Navigates to Admin → Jobs from sidebar                   | Job Status Dashboard loads. Shows summary cards: **Running (2)**, **Queued (5)**, **Failed (1)**, **Completed today (23)**       | 😐 Neutral — sees the overview            |
| 2    | Notices "Failed (1)" card is highlighted with red status | Scrolls to Job Table below. Sees the failed job: `crm.export-contacts` — Failed at 03:47 AM — "Timeout after 30s"                | 😤 Concerned — but at least he can see it |
| 3    | Clicks the failed job row to expand detail               | Sees: Job ID, plugin source (CRM), schedule (`0 0 * * *`), last run time, error message, retry count (3/3), next retry scheduled | 🤔 Investigating                          |
| 4    | Clicks "Retry Now" button                                | Job re-queued. Status changes to "Queued". Toast: "Job crm.export-contacts re-queued successfully"                               | 😊 Relieved — took action                 |
| 5    | Checks back 5 minutes later                              | Job status is now "Completed" with completion time                                                                               | 🎉 Satisfied — issue resolved             |

**Outcome**: Marco identified a failed job, understood the error, and manually retried it — all without contacting support. Per FR-007, FR-008, FR-009.

**Delight opportunities**:

- **Step 1**: Real-time counter updates via SSE/WebSocket — no need to refresh
- **Step 2**: Failed jobs auto-sorted to top of the table

### Journey 5: Priya — Integrating Search in a Plugin

**Trigger**: Priya is building the CRM plugin and needs to make contacts searchable via the global search bar. She consults the Search Service API.

**Preconditions**:

- Priya has access to the Plexica developer documentation
- Search Service API is available (per FR-011, FR-012)
- CRM plugin is registered and active

#### Happy Path

| Step | User Action                                                                                                          | System Response                                                                                            | Emotional State                    |
| ---- | -------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------- |
| 1    | Reads Search Service docs: `POST /api/v1/search/index`                                                               | Docs explain the `Indexable` interface: `{ id, type, title, body, metadata }` with tenant_id auto-injected | 😐 Neutral — reading docs          |
| 2    | Implements indexing in CRM plugin: `search.index({ type: 'crm:contact', title: contact.name, body: contact.notes })` | Document indexed. Search API returns 200                                                                   | 😊 Straightforward API             |
| 3    | Tests search from the global search bar: types contact name                                                          | Contact appears in results under "crm:contact" type grouping                                               | 🎉 Satisfied — it works end-to-end |
| 4    | Indexes 500 contacts in batch                                                                                        | All indexed within 5 seconds                                                                               | 🎉 Confident in performance        |

**Outcome**: Priya integrated search in ~30 minutes. Her plugin's data now appears in the global search bar for all tenant users. Per FR-011, FR-013, FR-014.

**Delight opportunities**:

- **Step 1**: Interactive API playground in docs for testing search queries
- **Step 3**: Type icon customization — plugin can register a custom icon for its search result type

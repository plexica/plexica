## 2. User Journeys

---

### Journey 1: Laura — Workspace Setup and Member Invitation

**Trigger**: Laura's organization just onboarded to Plexica. She needs to
create a workspace structure that mirrors her org chart and invite team
members with appropriate roles.

**Preconditions**:

- Laura is logged in as a Tenant Admin
- The tenant exists with a Keycloak realm provisioned (Spec 002)
- No workspaces exist yet (first-time setup)

#### Happy Path

| Step | User Action                                          | System Response                                                                                                       | Emotional State             |
| ---- | ---------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- | --------------------------- |
| 1    | Laura sees the dashboard with no workspaces          | Dashboard shows empty state: "No workspaces yet. Create your first workspace to get started."                         | Neutral — guided            |
| 2    | Clicks "Create Workspace" CTA from empty state       | Create Workspace dialog opens with name, description, optional parent selector, and template selector                 | Curious — starting setup    |
| 3    | Names it "Engineering", picks "Team" template, saves | Workspace created with pre-configured child workspaces (from template). Toast: "Workspace created"                    | Satisfied — fast setup      |
| 4    | Workspace selector now shows "Engineering"           | Top nav workspace selector updates to "Engineering"; workspace context loads; tabs show Overview / Members / Settings | Oriented — knows where I am |
| 5    | Clicks "Members" tab                                 | Members list shown (only Laura as Admin). "Add Member" button visible.                                                | Ready to invite             |
| 6    | Clicks "Add Member", searches for "marco"            | Autocomplete shows no results (Marco not in tenant yet). Input switches to email invite mode.                         | Clear — knows next step     |
| 7    | Types Marco's email, selects "Member" role, sends    | Invitation sent. Toast: "Invitation sent to marco@company.com". Pending invite appears in list.                       | Satisfied — done            |
| 8    | Navigates to child workspace "Frontend Team"         | Workspace detail loads via workspace selector hierarchy                                                               | Oriented — hierarchy works  |
| 9    | Adds another member to child workspace               | Member added directly (already in tenant). Toast: "Member added."                                                     | Delighted — smooth flow     |

**Outcome**: Laura has created a workspace hierarchy and invited/added team
members. The organization structure is live and team members can access their
assigned workspaces.

**Delight opportunities**:

- Template auto-creates child workspaces, saving manual setup (step 3)
- Autocomplete for existing users avoids redundant email invites (step 6)
- Workspace selector provides instant context switching (step 8)

#### Edge Case A: Workspace Creation Fails (Duplicate Name)

**Trigger**: Laura creates a workspace with a name whose slug collides with
an existing workspace.

| Step | User Action                                  | System Response                                                                 | Emotional State     |
| ---- | -------------------------------------------- | ------------------------------------------------------------------------------- | ------------------- |
| 1    | Opens Create Workspace dialog                | Dialog shown                                                                    | Neutral             |
| 2    | Types "Engineering" (slug already taken)     | Real-time slug preview shows "engineering" — matches existing                   | Unaware             |
| 3    | Clicks "Create"                              | Inline error: "A workspace with this URL already exists. Try a different name." | Mildly frustrated   |
| 4    | Changes name to "Engineering v2" and creates | Workspace created with slug "engineering-v2"                                    | Relieved — easy fix |

**Recovery path**: Inline error on the name field. User modifies name. No
data loss.
**Feedback**: "A workspace with this URL already exists. Try a different name."

### Journey 2: Marco — Workspace Navigation and Profile Update

**Trigger**: Marco received an email invitation, accepted it, and now logs in
for the first time. He needs to find his workspace and set up his profile.

**Preconditions**:

- Marco has accepted a workspace invitation and has a Keycloak account
- Marco has Member role in "Engineering > Frontend Team"
- Marco has no other workspace roles

#### Happy Path

| Step | User Action                                         | System Response                                                                                     | Emotional State            |
| ---- | --------------------------------------------------- | --------------------------------------------------------------------------------------------------- | -------------------------- |
| 1    | Marco logs in for the first time                    | Dashboard shows: "Welcome, Marco". Workspace selector shows "Frontend Team" (his only workspace)    | Curious — new platform     |
| 2    | Clicks workspace selector, sees "Frontend Team"     | Selector dropdown shows his accessible workspaces with hierarchy path "Engineering > Frontend Team" | Oriented — clear context   |
| 3    | Selects "Frontend Team"                             | Workspace detail loads: Overview tab with workspace description and recent activity                 | Comfortable — familiar     |
| 4    | Explores Members tab                                | Sees list of team members with roles. No "Add Member" button (Marco is Member, not Admin).          | Informed — understands     |
| 5    | Tries to access Settings tab                        | Tab is visible but content shows: "Only workspace admins can modify settings."                      | Neutral — clear boundary   |
| 6    | Clicks avatar in header, selects "Profile"          | Profile page opens: display name, avatar upload, timezone, language, notification preferences       | Ready to personalize       |
| 7    | Updates display name, uploads avatar, sets timezone | All changes saved. Toast: "Profile updated". Avatar reflects in header immediately.                 | Delighted — instant update |

**Outcome**: Marco has navigated to his workspace, understands his access
level, and has personalized his profile. He's ready to start working.

**Delight opportunities**:

- Workspace selector auto-selects the only available workspace (step 2)
- Profile changes reflect instantly across the UI (step 7)
- Hierarchy breadcrumb in workspace selector clarifies nested structure

#### Edge Case A: Permission Denied on Workspace

**Trigger**: Marco tries to directly access a workspace URL he doesn't have
access to (e.g., typing the URL or following a shared link).

| Step | User Action                        | System Response                                                                               | Emotional State |
| ---- | ---------------------------------- | --------------------------------------------------------------------------------------------- | --------------- |
| 1    | Navigates to `/workspaces/some-id` | API returns 403/404. Content area shows: "You don't have access to this workspace."           | Confused        |
| 2    | Reads the message                  | Message includes: "Contact your workspace admin to request access." and "Go to Dashboard" CTA | Informed        |
| 3    | Clicks "Go to Dashboard"           | Redirected to dashboard with his accessible workspaces                                        | Redirected — ok |

**Message**: "You don't have access to this workspace. Contact your workspace
admin to request access."

#### Edge Case B: Avatar Upload Too Large

| Step | User Action                       | System Response                                                                                          | Emotional State   |
| ---- | --------------------------------- | -------------------------------------------------------------------------------------------------------- | ----------------- |

### Journey 3: Sofia — Audit Log Review and Branding Setup

**Trigger**: Sofia needs to check what changes were made in her organization
over the past week, and set up her tenant's branding (logo and colors).

**Preconditions**:

- Sofia is a Tenant Admin
- The tenant has active workspaces with recent activity
- No branding has been configured yet

#### Happy Path

| Step | User Action                                         | System Response                                                                                 | Emotional State               |
| ---- | --------------------------------------------------- | ----------------------------------------------------------------------------------------------- | ----------------------------- |
| 1    | Sofia navigates to Settings via sidebar             | Tenant Settings page loads with tabs: General / Branding / Auth / Audit Log                     | Oriented — clear structure    |
| 2    | Clicks "Audit Log" tab                              | Table loads with recent events: columns for Date, User, Action, Target, Details                 | Curious — scanning data       |
| 3    | Filters by action type "Membership" and last 7 days | Table updates to show only member additions/removals/role changes in the last week              | Focused — finding what I need |
| 4    | Clicks on a row to see details                      | Expandable row reveals: before/after values, IP address, exact timestamp                        | Satisfied — sufficient detail |
| 5    | Navigates to "Branding" tab                         | Branding page: logo upload, primary color picker, dark mode toggle. Current values: default     | Ready to customize            |
| 6    | Uploads organization logo (PNG, 800KB)              | Upload succeeds. Logo preview shows on the page. Header/sidebar logo area updates in real-time. | Delighted — instant preview   |
| 7    | Picks a primary color (#2D5F2D, forest green)       | Color picker shows the selection. UI elements (buttons, links, active states) preview update.   | Impressed — live preview      |
| 8    | Clicks "Save"                                       | Branding saved. Toast: "Branding updated." Entire UI reflects new color scheme.                 | Delighted — professional look |

**Outcome**: Sofia has reviewed audit data to understand recent changes and
branded the platform to match her organization's identity.

**Delight opportunities**:

- Live branding preview before saving (step 7)
- Expandable audit log rows for detail without navigation (step 4)
- Filter presets for common audit queries ("Last 7 days", "Membership changes")

#### Edge Case C: Auth Realm Lockout Prevention

**Trigger**: Sofia (as Tenant Admin) tries to disable all authentication

| Step | User Action                                     | System Response                                                                                | Emotional State      |

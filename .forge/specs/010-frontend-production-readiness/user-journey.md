## 2. User Journeys

---

### Journey 1: Dana — Seeing Tenant Branding Applied

**Trigger**: Dana has just uploaded her company's logo and set brand colors in
the Tenant Admin panel (out of scope for this spec — covered by Spec 008).
She now opens the main application to verify the branding is applied.

**Preconditions**:

- Dana is logged in as a tenant administrator
- Tenant theme has been configured via `PUT /api/v1/tenant/settings` with logo URL, primary color `#1B5E20` (dark green), heading font "Poppins", body font "Lato"
- The tenant has at least one plugin installed (e.g., CRM)

#### Happy Path

| Step | User Action                                | System Response                                                                                                                                   | Emotional State                                   |
| ---- | ------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------- |
| 1    | Dana opens the platform URL in her browser | Shell loads, ThemeProvider fetches `/api/v1/tenant/settings`                                                                                      | 😐 Neutral — waiting for page to load             |
| 2    | Dana sees the header area                  | Her company logo appears in the top-left corner (max 40px height); header uses default background color                                           | 😊 Pleased — "It's our logo!"                     |
| 3    | Dana scans the page content                | Primary buttons and links use dark green (`#1B5E20`); headings use Poppins font; body text uses Lato font                                         | 🎉 Satisfied — "This looks like our tool now"     |
| 4    | Dana navigates to the CRM plugin page      | Plugin loads within its error boundary; all themed elements (buttons, links, inputs) inherit the tenant's dark green color and Poppins/Lato fonts | 😊 Relieved — "Even the plugin matches our brand" |
| 5    | Dana opens the sidebar navigation          | Sidebar active item uses the tenant's primary color as a left-border accent                                                                       | 🎉 Delighted — consistent branding throughout     |

**Outcome**: Dana sees a fully branded experience. Her company logo is in the
header, brand colors apply to all primary UI elements, and custom fonts render
throughout the application — including within plugin areas.

**Delight opportunities**:

- Instant theme application on page load (< 50ms, no flash of unstyled content per NFR-014)
- Fonts load with `display: swap` so text is visible immediately with system fonts, then swaps seamlessly to brand fonts
- Logo is clickable and links to home (`/`)

#### Edge Case A: Theme API Returns 404 (No Theme Configured)

**Trigger**: Tenant has not configured any custom theme. `GET /api/v1/tenant/settings` returns `settings.theme: null`.

| Step | User Action             | System Response                                                                      | Emotional State                 |
| ---- | ----------------------- | ------------------------------------------------------------------------------------ | ------------------------------- |
| 1    | Dana opens the platform | ThemeProvider fetches settings, gets null theme                                      | 😐 Neutral                      |
| 2    | Dana sees the header    | Default Plexica wordmark logo appears                                                | 😐 Neutral — expected           |
| 3    | Dana scans the page     | Default color palette (blue `#0066CC` primary), Inter heading font, Roboto body font | 😐 Neutral — looks professional |

**Recovery path**: No recovery needed. The default Plexica theme provides a
complete, accessible, professional appearance. No error is shown.

**Feedback**: None — the default theme applies silently. No "theme not found"
messages are shown to the user.

---

#### Edge Case B: Tenant Logo URL Returns 404 (Broken Image)

**Trigger**: Tenant theme has a `logo` URL that is unreachable (e.g., CDN file was deleted).

| Step | User Action                                 | System Response                                                                    | Emotional State                            |
| ---- | ------------------------------------------- | ---------------------------------------------------------------------------------- | ------------------------------------------ |
| 1    | Dana opens the platform                     | ThemeProvider fetches settings, gets valid theme with broken logo URL              | 😐 Neutral                                 |
| 2    | Dana sees the header                        | `<img>` fires `onError`; logo automatically falls back to default Plexica wordmark | 😕 Slightly confused — "Where's our logo?" |
| 3    | Dana checks browser console (if tech-savvy) | Warning logged: `"Tenant logo failed to load: 404 Not Found"`                      | 😐 Informed                                |

**Recovery path**: Dana can re-upload the logo in Tenant Admin settings (Spec 008).
The shell continues to work normally with the default logo.

**Feedback**: No user-facing error message. The fallback is silent — the
Plexica wordmark replaces the broken logo automatically.

---

#### Edge Case C: Custom Font Fails to Load (CDN Down)

**Trigger**: Self-hosted font files are unreachable (MinIO/CDN outage).

| Step | User Action                          | System Response                                                                                                                                                                     | Emotional State                                         |
| ---- | ------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------- |
| 1    | Dana opens the platform              | ThemeProvider fetches settings; font-loader calls `FontFace.load()` which rejects                                                                                                   | 😐 Neutral                                              |
| 2    | Dana sees the page                   | System fallback fonts render (e.g., `system-ui, sans-serif`). Colors apply correctly. No layout shift because `font-display: swap` is used and fallback fonts have similar metrics. | 😐 Neutral — text looks slightly different but readable |
| 3    | Dana refreshes later (CDN recovered) | Custom fonts load successfully; full branding restored                                                                                                                              | 😊 Relieved                                             |

**Recovery path**: Automatic. `font-display: swap` ensures text is always
readable. Font-loader catches the error and logs a warning. No retry loop.

**Feedback**: Warning in console: `"Font loading failed; system fonts in use"`.
No user-facing error.

### Journey 2: Raj — Developing a Widget for Cross-Plugin Use

**Trigger**: Raj has built a `ContactCard` widget in his CRM plugin and wants
to verify that another plugin can load and render it.

**Preconditions**:

- Raj is logged in as a developer on a development tenant
- CRM plugin is installed and running with `ContactCard` exposed in Module Federation config
- Raj has access to a second plugin (e.g., "Sales Dashboard") where he'll import the widget

#### Happy Path

| Step | User Action                                                                                              | System Response                                                                                        | Emotional State                                           |
| ---- | -------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ | --------------------------------------------------------- |
| 1    | Raj adds `loadWidget({ pluginId: 'crm', widgetName: 'ContactCard' })` to his Sales Dashboard plugin code | N/A (development action)                                                                               | 😐 Neutral                                                |
| 2    | Raj navigates to the Sales Dashboard plugin in the browser                                               | Shell loads the Sales Dashboard route within its error boundary                                        | 😐 Neutral                                                |
| 3    | Raj scrolls to the section that embeds the ContactCard widget                                            | Suspense fallback shows a brief loading skeleton while `WidgetLoader` fetches the CRM remote entry     | 🤔 Curious — "Is this going to work?"                     |
| 4    | Widget loads successfully                                                                                | `ContactCard` renders with contact name, email, phone — inheriting the tenant's theme colors and fonts | 🎉 Delighted — "It just works!"                           |
| 5    | Raj passes different `contactId` props                                                                   | Widget updates reactively, showing different contact data                                              | 😊 Satisfied — "It behaves like a normal React component" |

**Outcome**: The CRM widget renders inside another plugin's page. It inherits
the tenant theme, responds to props, and is isolated by the error boundary.

**Delight opportunities**:

- Widget load time < 300ms (NFR-015) — feels instant
- Widget inherits theme automatically — no additional code needed by widget consumers
- TypeScript types for widget props provide autocomplete in IDE

#### Edge Case A: Widget Remote Unavailable (Plugin Not Running)

**Trigger**: The CRM plugin container is stopped or the remote entry URL is
unreachable.

| Step | User Action                                      | System Response                                                                                                             | Emotional State                |
| ---- | ------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------- | ------------------------------ |
| 1    | Raj navigates to Sales Dashboard                 | Shell loads successfully                                                                                                    | 😐 Neutral                     |
| 2    | WidgetLoader attempts to fetch `crm/ContactCard` | Module Federation import fails (network error)                                                                              | —                              |
| 3    | Raj sees the widget area                         | WidgetFallback placeholder appears: gray dashed-border box with "📦 Widget Unavailable" and "crm/ContactCard" in muted text | 😕 Slightly frustrated         |
| 4    | Raj checks the rest of the page                  | All other content renders normally; the Sales Dashboard is fully functional except for the CRM widget embed                 | 😊 Relieved — not a full crash |

**Recovery path**: Raj starts the CRM plugin container. On next page navigation
(or refresh), the widget loads successfully. No automatic retry for widgets
(per spec — auto-retries can be noisy).

**Feedback**: Console logs: `"Failed to load widget: crm/ContactCard — Error: Failed to fetch module"`.
User-facing: dashed-border placeholder with minimal copy.

#### Edge Case B: Widget Version Mismatch

**Trigger**: Shell expects React 19 singleton, CRM plugin was built with
React 18. Module Federation shared config detects version incompatibility.

| Step | User Action                                               | System Response                                                                                                    | Emotional State                       |
| ---- | --------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------ | ------------------------------------- |
| 1    | Raj navigates to Sales Dashboard with embedded CRM widget | Module Federation attempts to resolve shared React singleton                                                       | —                                     |
| 2    | Version mismatch detected                                 | If incompatible: runtime error caught by `PluginErrorBoundary` wrapping the widget                                 | 😤 Frustrated                         |
| 3    | Raj sees the widget area                                  | Error boundary fallback: "⚠️ Plugin Unavailable — The CRM plugin could not be loaded." with Retry button           | 😕 Confused but informed              |
| 4    | Raj checks console                                        | Structured error log with `pluginId: "crm"`, `widgetName: "ContactCard"`, and stack trace showing version conflict | 🤔 Debugging — "Ah, version mismatch" |

**Recovery path**: Raj updates his CRM plugin to use React 19 and redeploys.
The Retry button re-attempts the widget load.

**Feedback**: Error boundary provides the plugin name and a user-friendly
message. Console log provides the technical details.

### Journey 3: Maria — Encountering a Plugin Error

**Trigger**: Maria clicks on the CRM plugin in the sidebar navigation. The
plugin has a bug in its render method (introduced in a recent update).

**Preconditions**:

- Maria is logged in as a regular user
- CRM plugin is installed and the route exists in the sidebar
- CRM plugin component throws an error during rendering

#### Happy Path (Error Recovery)

| Step | User Action                                     | System Response                                                                                                                                                                                                                   | Emotional State                               |
| ---- | ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------- |
| 1    | Maria clicks "CRM" in the sidebar               | Router navigates to `/plugins/crm`; Plugin error boundary wraps the lazy-loaded component                                                                                                                                         | 😐 Neutral                                    |
| 2    | Plugin component throws an error during render  | `PluginErrorBoundary.componentDidCatch` fires; error context logged to console                                                                                                                                                    | —                                             |
| 3    | Maria sees the error fallback                   | Centered card appears: "⚠️ Plugin Unavailable — The 'CRM' plugin could not be loaded. This might be a temporary network issue." Error message (simplified) shown below. Two buttons: [Retry] (primary) and [Go Back] (secondary). | 😤 Frustrated — but informed                  |
| 4    | Maria reads the message                         | She understands it's a plugin issue, not a platform-wide failure                                                                                                                                                                  | 😐 Reassured — "OK, it's just this one thing" |
| 5    | Maria clicks [Go Back]                          | Browser navigates back to previous page (e.g., Dashboard)                                                                                                                                                                         | 😊 Relieved — rest of app works               |
| 6    | Maria tries again later (after plugin is fixed) | CRM loads successfully                                                                                                                                                                                                            | 🎉 Satisfied                                  |

**Outcome**: Maria's session is preserved. She can continue using other parts
of the platform. The error is contained to the CRM plugin area.

**Delight opportunities**:

- Header and sidebar remain fully functional during the error
- "Go Back" button provides an immediate escape
- Error message uses plain language, not technical jargon
- Plugin name is shown in quotes for clarity ("CRM" not "crm-v2.1.3")

#### Edge Case A: Plugin Remote Entry Unreachable (Network Error)

**Trigger**: Maria's network is degraded, or the plugin CDN is down. The
`remoteEntry.js` file fails to fetch.

| Step | User Action                                 | System Response                                                                                                            | Emotional State                |
| ---- | ------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------- | ------------------------------ |
| 1    | Maria clicks "CRM" in sidebar               | Router navigates to `/plugins/crm`                                                                                         | 😐 Neutral                     |
| 2    | `React.lazy` import fails                   | Network error caught; error boundary activates                                                                             | —                              |
| 3    | Maria sees the error fallback               | "⚠️ Plugin Unavailable — The 'CRM' plugin could not be loaded. This might be a temporary network issue." [Retry] [Go Back] | 😤 Frustrated                  |
| 4    | Maria clicks [Retry]                        | Error boundary resets; `React.lazy` re-attempts the import                                                                 | 🤔 Hopeful                     |
| 5a   | (If network recovered) Plugin loads         | CRM page renders normally                                                                                                  | 🎉 Relieved                    |
| 5b   | (If still down) Error boundary re-activates | Same fallback shown again                                                                                                  | 😡 Frustrated — "Still broken" |

**Recovery path**: The [Retry] button resets the error boundary, which
re-mounts the children and triggers a fresh `React.lazy` import attempt.

**Feedback**: "This might be a temporary network issue" — guides the user to
wait and retry rather than contacting support.

#### Edge Case B: Root Error Boundary (Shell Crash)

**Trigger**: An unexpected error occurs in the shell itself (outside of any
plugin boundary) — e.g., a bug in the ThemeProvider or Router.

| Step | User Action                       | System Response                                                                                                                               | Emotional State                        |
| ---- | --------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------- |
| 1    | Maria opens the platform          | Shell component tree throws during render                                                                                                     | —                                      |
| 2    | Root error boundary catches       | Full-page error screen: centered layout with Plexica logo, "Something went wrong" heading, brief apologetic message, and [Reload Page] button | 😡 Frustrated — "Everything is broken" |
| 3    | Maria clicks [Reload Page]        | `window.location.reload()` triggered; full page refresh                                                                                       | 🤔 Hopeful                             |
| 4a   | (If transient) App loads normally | Maria continues working                                                                                                                       | 😊 Relieved                            |
| 4b   | (If persistent) Same error        | Error page shown again; includes "If this keeps happening, contact support at support@plexica.io"                                             | 😡 Frustrated but has next step        |

**Recovery path**: Full page reload. If error persists, support contact info
is provided.

**Feedback**: Root error page is distinct from plugin error boundary — it's a
full-page layout (no header/sidebar) because those components may be part of
the crash. Includes support contact for persistent issues.

#### Edge Case C: Maria Uses Browser Zoom at 150%

**Trigger**: Maria uses the platform at 150% zoom due to low vision. Per
WCAG 2.1 1.4.4, text must be resizable to 200% without loss.

| Step | User Action                                                 | System Response                                                                                                                                      | Emotional State                     |
| ---- | ----------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------- |
| 1    | Maria opens the platform at 150% zoom                       | Layout adapts; responsive breakpoints may trigger tablet layout at lower effective viewport                                                          | 😐 Neutral                          |
| 2    | Maria sees an error boundary fallback (if a plugin crashes) | Fallback card is centered, max-width constrained; text reflows naturally; [Retry] and [Go Back] buttons are large enough to tap (≥44×44px effective) | 😐 Neutral — everything is readable |
| 3    | Maria navigates normally                                    | All interactive elements are visible; no horizontal scrolling needed; focus indicators are visible                                                   | 😊 Relieved — "I can use this"      |

**Recovery path**: N/A — this is a normal usage pattern, not an error.

**Feedback**: The design uses relative units (`rem`, `%`), flexible layouts
(`flex`, `grid`), and constrained max-widths to ensure usability at 200% zoom.

### H-001 — HIGH | Test-Spec Coherence | CI Blocker

**File**: `apps/web/src/__tests__/layout-engine/VisibilityToggle.test.tsx`  
**Lines**: Multiple (all 6 ARIA label `expect` assertions)

**Description**:  
The ARIA label assertions in `VisibilityToggle.test.tsx` do not match the string actually generated by the component. The component (`VisibilityToggle.tsx`) builds the label as:

```
`Set ${field.label} visibility for ${role} to read-only. Currently ${currentValue === 'readonly' ? 'read-only' : 'editable'}`
```

But the tests assert the format:

```
`${field.label} visibility for ${role}: ${currentValue === 'readonly' ? 'read-only' : 'editable'}`
```

For example, for field `"Budget"`, role `"ADMIN"`, value `"editable"`:

- **Component produces**: `"Set Budget visibility for ADMIN to read-only. Currently editable"`
- **Test asserts**: `"Budget visibility for ADMIN: editable"`

All 6 test cases covering ARIA labels will fail. This is a **CI blocker** — these tests cannot pass in their current form.

**Required fix**:  
Update all ARIA label assertions in `VisibilityToggle.test.tsx` to match the exact string format produced by `VisibilityToggle.tsx`. Alternatively, if the component format is wrong, fix the component — but the test and component must agree.

**Spec reference**: NFR-006 (WCAG 2.1 AA); the ARIA label tests exist specifically to enforce accessibility compliance.

### M-001 — MEDIUM | Correctness | Stale Closure

**File**: `apps/web/src/components/layout-engine/LayoutConfigPanel.tsx`

**Description**:  
`useEffect` on mount has an empty dependency array with an `eslint-disable-next-line react-hooks/exhaustive-deps` suppression comment. Inside the effect, `loadForms()` is called. `loadForms` is a `useCallback` that depends on `selectedFormId`. Because the effect never re-runs, if `selectedFormId` changes from an external source (e.g., URL param, parent state) after initial mount, `loadForms` will execute with a stale closure referencing the initial `selectedFormId` value.

**Impact**: Form list may not reload when the active form changes through navigation. In practice this is partially mitigated by the React Query cache, but the stale closure means manual invalidation paths could silently do nothing.

**Recommended fix**: Remove the `eslint-disable` suppression and add `loadForms` to the effect dependency array, or restructure to use React Query's `enabled`/`queryKey` mechanism so the query reruns automatically when `selectedFormId` changes.

### M-002 — MEDIUM | Maintainability | Suppressed Exhaustive-Deps

**File**: `apps/web/src/components/layout-engine/LayoutConfigPanel.tsx`

**Description**:  
`handleSave` useCallback has `eslint-disable-next-line react-hooks/exhaustive-deps` suppressing `buildSaveInput` from the dependency list. `buildSaveInput` references `fieldConfigs`, `columnConfigs`, `sectionConfigs`, and `formId` — all of which are state/props that could change. If any of these change between renders without triggering `handleSave` to be recreated, the save callback will use stale values.

**Recommended fix**: Either include `buildSaveInput` in the `useCallback` deps (it should be a `useCallback` itself, or a `useMemo`), or inline `buildSaveInput` directly into `handleSave` to avoid the closure issue.

### M-003 — MEDIUM | Correctness | Section Grouping Non-Functional

**File**: `apps/web/src/components/layout-engine/LayoutAwareForm.tsx`

**Description**:  
In element-children mode (when `children` is passed as React elements rather than a render prop), the component filters `renderedFields` by `data-section-id` attribute to group fields into sections:

```tsx
const sectionFields = renderedFields.filter((el) => el.props['data-section-id'] === section.id);
```

However, the component never attaches `data-section-id` to the cloned/wrapped child elements. The `React.cloneElement` call on line ~403 only spreads `layoutProps` (visibility, readOnly, label overrides) — it does not inject `data-section-id`. As a result, `sectionFields` will always be empty for every section, and all fields will fall into the unsectioned catch-all block.

**Impact**: FR-004 (section grouping and ordering) is silently non-functional when using element-children mode. This is a regression — the render-prop mode works correctly because the prop is passed through the render function signature.

**Spec reference**: FR-004 — "Users can reorder and rename form sections."

**Required fix**:

```tsx
// In the cloneElement call, inject data-section-id alongside layoutProps:
React.cloneElement(child, {
  ...layoutProps,
  'data-section-id': child.props['data-section-id'], // preserve existing
});
```

Or, more robustly, maintain an internal `fieldToSection` map built from the layout config and use that for grouping rather than relying on `data-section-id` being pre-attached.

### M-004 — MEDIUM | UX Quality | ARIA Role Conflict

**File**: `apps/web/src/components/layout-engine/RolePreviewPanel.tsx`

**Description**:  
The preview card container element uses `role="presentation"` to suppress its implicit landmark semantics. However, the container contains `<input>` and `<label>` elements which are inherently interactive. `role="presentation"` is prohibited on elements with focusable descendants — assistive technologies (NVDA, JAWS, VoiceOver) will exhibit inconsistent behaviour: some will announce the inputs normally, others will suppress all semantics including the labels, making the preview inputs inaccessible.

**Recommended fix**: Remove `role="presentation"` from the preview card container. If the intent is to suppress a redundant landmark, use `aria-hidden="true"` on a purely decorative wrapper that does not contain any interactive elements. The interactive preview elements should remain in the normal accessibility tree.

**Spec reference**: NFR-006 (WCAG 2.1 AA).

### M-005 — MEDIUM | Security | Silent Workspace→Tenant Scope Fallthrough

**File**: `apps/web/src/components/layout-engine/LayoutConfigPanel.tsx`

**Description**:  
When `scope='workspace'` is passed but `scopeId` is `undefined` or `null`, the component silently falls through to the tenant-scope API call path (the `if (!scopeId)` guard returns early from the workspace branch and execution continues to the tenant path). There is no error thrown, no console warning, and no TypeScript discriminated union preventing this invalid combination.

A workspace admin opening the layout config panel for a specific workspace, with `scopeId` temporarily undefined due to a loading race, could inadvertently load and then save over the tenant-level layout config.

**Recommended fix**:

```tsx
// Option A: TypeScript discriminated union (preferred)
type LayoutConfigPanelProps =
  | { scope: 'tenant'; scopeId?: never }
  | { scope: 'workspace'; scopeId: string };

// Option B: Runtime guard with explicit error
if (scope === 'workspace' && !scopeId) {
  throw new Error('LayoutConfigPanel: scopeId is required when scope is "workspace"');
}
```

**Description**:

**File**: `apps/web/src/components/layout-engine/LayoutConfigPanel.tsx`

**Description**:

This means `buildSaveInput` is recreated on every render and its identity is unstable, which is why the lint rule had to be suppressed in M-002. Promoting it to `useCallback` or `useMemo` would eliminate the need for the lint suppression.

### L-004 — LOW | Test-Spec Coherence | `null` vs `undefined` in Loading State

**File**: `apps/web/src/__tests__/layout-engine/useResolvedLayout.test.tsx`  
**Line**: ~124

**Description**:
